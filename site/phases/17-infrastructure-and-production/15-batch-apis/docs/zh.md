# 批处理 API — 50% 折扣成为行业标准

> 每个主要供应商都提供异步批处理 API，折扣 50%，周转约 24 小时。OpenAI、Anthropic、Google 及大多数推理平台（Fireworks 批处理层、Together 批处理）都实现了相同模式。叠加提示词缓存和隔夜流水线，降至同步无缓存成本的约 10%。规则残酷简单：非交互式的都应该走批处理。内容生成流水线、文档分类、数据提取、报告生成、批量标注、目录打标——任何能容忍 24 小时延迟的都是省钱的机会，直到它上了批处理。2026 年生产模式：每个新 LLM 工作负载三车道分诊：交互式（同步加缓存）、半交互式（异步队列加回退）、批处理（隔夜，缓存输入叠加）。假装交互式但能耐受分钟级延迟的工作负载浪费最多。

**类型：** 精读
**语言：** Python（标准库，玩具级批处理 vs 同步成本模拟器）
**前置要求：** Phase 17 · 14（提示词和语义缓存）
**时长：** 约 45 分钟

## 学习目标

- 说出三个供应商批处理 API（OpenAI、Anthropic、Google）及共同的 50% 折扣加 24 小时周转保证。
- 计算隔夜分类工作负载叠加批处理 + 缓存输入的成本，与同步无缓存基线对比。
- 将工作负载分诊为交互式/半交互式/批处理，并为每条说明理由。
- 说出两个陷阱：部分交互性（用户期待快于 24h）和输出 schema 漂移（各供应商批处理文件格式不同）。

## 背景问题

你的团队上线了一个隔夜报告生成流水线。5 万份文档，每份摘要，聚类摘要，起草高管简报。同步跑要 4 小时，$2,000/晚。你听说了批处理 API。

批处理给你 50% 折扣。你还启用了系统提示词的提示词缓存（5 万条调用共享）。叠加后账单降到 $180/晚——约基线的 9%。同一流水线，三个配置变更。

批处理是 LLM 成本工具箱里没人拉的那个最便宜的杠杆。原因主要是组织性的：团队想到"实时"但 SLA 实际是"明早"。本课是关于别把账单 90% 落在桌上。

## 核心概念

### 三个批处理 API

**OpenAI Batch API**：JSONL 文件上传，含请求列表。承诺 24 小时周转（实际通常约 2-8 小时）。输入输出 token 均 50% 折扣。`/v1/batches` 端点。可缓存输入同样享受缓存输入价格。

**Anthropic Message Batches**：JSONL 上传。24 小时周转。50% 折扣。支持 `cache_control`——写是显式的，在批处理内自动发生读取。

**Google Vertex AI Batch Prediction**：BigQuery 或 GCS 输入。Gemini 类似 50% 折扣。与 Vertex 流水线集成。

### 语义：异步，不是慢

批处理是"我承诺 24 小时内返回"——不是"这要花 24 小时"。P50 通常 2-6 小时。供应商在 GPU 库存利用不足的峰谷时段调度你的批处理。

### 叠加缓存

5 万文档摘要，相同 4K-token 系统提示词：

- 同步无缓存：50000 ×（$input × 4000 + $output × 200）全价。
- 同步缓存：系统提示词首次写后缓存；剩余 49999 条享受 10 倍便宜输入。
- 批处理缓存：上述全部再加输入读取和写入 50% 折扣。

叠加：批处理 + 缓存 = 约同步无缓存账单的 10%。任何隔夜运行且有共享系统提示词的工作负载都应该用它。

### 工作负载分诊

**交互式** — 用户等待响应。TTFT 很重要。同步调用加提示词缓存。不能批处理。

**半交互式** — 用户提交任务，几分钟后回查。异步队列加回退到同步（批处理不可用时）。想中等容量 RAG 索引。

**批处理** — 用户期望"明早"或"下一小时"。内容流水线、大规模分类、离线分析。常驻批处理，常驻叠加缓存。

常见错误：把一切都分类为交互式因为流水线是生产级的。生产不是延迟规格——SLA 才是。

### 部分交互性陷阱

有些功能看起来是交互式的但能耐受 5-10 分钟。比如：带"刷新"按钮的夜间客户健康报告。用户点刷新；等 10 分钟没问题。团队把它做成了同步。50 个并发刷新花 10 倍于批处理加邮件发送的成本。

要问的问题："24 小时对这个用户意味着什么？"如果答案是"他们不会注意到"，走批处理。

### 输出 schema 陷阱

各供应商批处理文件格式不同：

- OpenAI：JSONL，每行一个请求。
- Anthropic：JSONL，每行一条消息；响应格式嵌入。
- Vertex：BigQuery 表或带 TFRecord 的 GCS 前缀。

写"一个跨供应商批处理客户端"意味着每个供应商的适配代码。宣传多供应商批处理的网关（Portkey、LiteLLM 部分层）底层仍是包装原始格式。

### 必须记住的数字

- 各供应商批处理折扣：输入+输出均 50%。
- 周转 SLA：保证 24 小时，典型 P50 2-6 小时。
- 叠加批处理 + 缓存输入：约同步无缓存成本的 10%。
- 工作负载分诊原则：能耐受 24 小时延迟，永远走批处理。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|----------|----------|
| 批处理 API | "异步折扣" | 50% 折扣，24 小时周转 |
| JSONL | "批处理格式" | 每行一个 JSON 请求；OpenAI/Anthropic 标准 |
| Message Batches | "Anthropic 批处理" | Anthropic 批处理 API 产品名 |
| Batch Prediction | "Vertex 批处理" | Vertex AI 批处理 API |
| 周转 SLA | "24 小时承诺" | 保证，不是典型值；典型 2-6h |
| 工作负载分诊 | "交互性决策" | 交互式/半交互式/批处理路由决策 |
| 输出 schema | "响应格式" | 各供应商 JSONL 布局；不可移植 |
| 叠加折扣 | "批处理 + 缓存" | 两者都适用时约无缓存同步账单的 10% |
