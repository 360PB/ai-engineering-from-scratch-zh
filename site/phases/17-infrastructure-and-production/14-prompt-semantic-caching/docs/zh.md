# 提示词缓存与语义缓存经济

> **定价快照日期 2026-04。** 以下数字引用了本课发布时的供应商价目表；向下游引用前请核实链接文档。

> 缓存发生在两个层级。L2（供应商级）提示词/前缀缓存复用注意力的 KV——Anthropic 的提示词缓存文档宣传长提示词成本最高削减 90%、延迟削减 85%；Claude 3.5 Sonnet 缓存读 $0.30/M vs 全新 $3.00/M，5 分钟 TTL，1 小时 TTL 选项写溢价 2 倍（docs.anthropic.com，2026-04）。OpenAI 提示词缓存自动应用于 ≥1024 token 的提示词，缓存输入约按全新 10% 折扣定价（platform.openai.com，2026-04）；具体每模型缓存价格取决于实时价目表。L1（应用级）语义缓存在 embedding 相似度命中时完全跳过 LLM。供应商"95% 准确率"指匹配正确性，不是命中率——报告的生产命中率从 10%（开放聊天）到 70%（结构化 FAQ）不等；两供应商均未发布官方基线，因此这些是社区遥测而非保证。生产坑：并行化破坏缓存（N 条并行请求在首次缓存写入完成前发出，可使支出膨胀数倍），以及前缀内的动态内容完全阻止缓存命中。ProjectDiscovery 报告通过将动态文本移出可缓存前缀，命中率从 7% 提升到 74%（2025-11）。

## 学习目标

- 区分 L2 提示词/前缀缓存（供应商处 KV 复用）和 L1 语义缓存（相似提示词完全绕过 LLM）。
- 解释 Anthropic 的 `cache_control` 显式标记和两个 TTL 选项（5 分钟 vs 1 小时）及其价格倍率。
- 给定命中率、提示词/响应配比和 token 价格，计算预期月节省。
- 说出使账单膨胀 5-10 倍的并行化反模式，以及使命中率崩溃的动态内容反模式。

## 背景问题

你给 RAG 服务加了提示词缓存。账单纹丝不动。你测量命中率；是 7%。你的提示词看似静态但其实不是——系统提示词包含精确到分钟的当前日期、一个请求 ID、以及为多样性而随机重排的示例。每次请求写一条新缓存条目，读取数为零。

另外，你的 Agent 每用户问题发起十个并行工具调用。十条请求在提供商看到提示词之前全到了。十次写入，零次读取。你的账单是"有缓存"该有的 5-10 倍。

缓存是一种协议，不是开关。两个层级，两种不同的失败模式。

## 核心概念

### L2 — 供应商提示词/前缀缓存

供应商存储可缓存前缀的注意力 KV，并在下一条匹配前缀的请求上复用。一次付写成本，读取几乎免费。

**Anthropic（Claude 3.5 / 3.7 / 4 系列）**：请求中的显式 `cache_control` 标记。你标出哪些块可缓存。TTL：5 分钟（写成本 1.25 倍基准）或 1 小时（写成本 2 倍基准）。缓存读取：Claude 3.5 Sonnet 上 $0.30/M vs 全新 $3.00/M——便宜 10 倍（docs.anthropic.com，2026-04 快照）。各模型费率不同（Opus/Haiku 分开发布）；请务必核查实时定价页。

**OpenAI**：自动缓存，适用于 ≥1024 token 的提示词（platform.openai.com，2026-04 快照）。无显式标记。在当前 gpt-4o/gpt-5 价目表上，缓存输入约全新 10%。docs 或 release notes 未发布官方命中率基线；社区报告在仔细提示词设计下聚在 30-60%。监控 `usage.cached_tokens` 测量自己的命中率。

**Google（Gemini）**：通过显式 API 做上下文缓存；100 万 token 上下文意味着缓存更划算。

**自托管（vLLM、SGLang）**：Phase 17 · 06 覆盖 RadixAttention——在自己计算上相同模式。

### L1 — 应用级语义缓存

完全在调用 LLM 之前，先哈希提示词，做 embedding，查找相似缓存请求（cosine 相似度高于阈值，通常 0.95+）。命中则返回缓存响应。未命中则调用 LLM 并缓存结果。

开源：Redis Vector Similarity、GPTCache、Qdrant。商业：Portkey Cache、Helicone Cache。

供应商准确率声称指的是返回的缓存响应在语义上合适的频率——不是命中率。生产命中率：

- 开放聊天：10-15%。
- 结构化 FAQ/支持：40-70%。
- 代码问题：20-30%（小变体扼杀命中）。
- 语音 Agent 重复提示词：50-80%（语音标准化固定集）。

### 并行化反模式

你的 Agent 发起 10 个并行工具调用。10 条都有相同 4K-token 系统提示词。Anthropic 缓存写入按请求计；首次缓存写入在提供商看到提示词约 300 ms 后完成。请求 2-10 同一毫秒窗口到达，每条都遇到缓存未命中。你付了 10 次写溢价，零次读取折扣。

修复：batch 加顺序优先——单独发出请求 1，然后在 1 的缓存填充后发出 2-10。首次工具调用增加 300 ms；节省 5-10 倍账单。

### 动态内容反模式

你的系统提示词像这样：

```
你是一个有用的助手。当前时间是 14:32:17。
用户 ID：abc123。今天是星期二...
```

每条请求都唯一。每条请求都写。零命中。

修复：把真正静态的内容移到可缓存前缀；动态内容追加到缓存边界之后：

```
[可缓存]
你是一个有用的助手。[规则、示例、指令]
[/可缓存]
[动态，不缓存]
当前时间：14:32:17。用户：abc123。
```

ProjectDiscovery 这样做将命中率从 7% 移到 74% 并公开发布了分析。

### 叠加批处理 + 缓存做夜间工作负载

批处理 API（Phase 17 · 15）给出 50% 折扣、24 小时周转。叠加缓存输入在此基础上再得约 10 倍。隔夜分类、标注和报告生成工作负载通过叠加可以降到同步无缓存基线的约 10%。

### 必须记住的数字

定价点是 2026-04 从链接供应商文档捕获，每隔几个月漂移——依赖前请重新核查。

- Anthropic 缓存读取：Claude 3.5 Sonnet 上 $0.30/M，约全新输入的 10 倍便宜（docs.anthropic.com）。
- Anthropic 缓存写溢价：1.25 倍（5 分钟 TTL）或 2 倍（1 小时 TTL）。
- OpenAI 自动缓存：适用于 ≥1024 token 提示词；在当前价目表上缓存输入约全新输入的 10%（platform.openai.com）。
- 语义缓存命中率（社区报告）：开放聊天约 10%；结构化 FAQ 高达约 70。不是供应商保证基线。
- ProjectDiscovery：动态移出前缀后命中率从 7% 到 74%（项目博客，2025-11）。
- 并行化反模式：在 N 条并行请求错过首次缓存写入时，账单通常膨胀 5-10 倍。

## 练习

1. 运行 `code/main.py`。切换并行化标志。账单变化多少？
2. 你的系统提示词包含日期。移出去。展示前后命中率数学。
3. 计算 1 小时 TTL（2 倍写）vs 5 分钟 TTL（1.25 倍写）的平衡点，给定你的请求到达率。
4. 语义缓存在 0.95 阈值命中 20%。在 0.85 时命中 50%，但你看到不正确的缓存响应。选正确的阈值并说明理由。
5. 你每用户问题 batch 10 个并行子查询。为缓存友好性重写，不增加端到端延迟。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|----------|----------|
| L2 提示词缓存 | "前缀缓存" | 供应商存储重复前缀的 KV |
| `cache_control` | "Anthropic 缓存标记" | 显式属性标记可缓存块 |
| 缓存写溢价 | "写税" | 首次未命中到缓存的额外成本（1.25 倍或 2 倍） |
| L1 语义缓存 | "embedding 缓存" | 调用 LLM 前先哈希+embedding 的应用级缓存 |
| GPTCache | "LLM 缓存库" | 流行的 OSS L1 缓存库 |
| 缓存命中率 | "命中/总数" | 从缓存服务的请求比例 |
| 并行化反模式 | "N 次写入陷阱" | N 条并行请求 N 次错过缓存 |
| 动态内容陷阱 | "时间在提示词中陷阱" | 前缀中动态字节杀死命中率 |
| RadixAttention | "同副本缓存" | SGLang 的前缀缓存实现 |
