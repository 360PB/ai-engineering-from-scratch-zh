# 工具接口 —— 为什么 Agent 需要结构化 I/O

> 语言模型生成 token，程序执行操作。两者之间的差距就是工具接口：一套合约，让模型可以请求操作，让宿主机执行操作。2026 年的每一套技术栈——OpenAI、Anthropic 和 Gemini 的函数调用；MCP 的 `tools/call`；A2A 的任务部件——都是对同一套四步循环的不同编码。本课命名这个循环，并展示运行它所需的最少组件。

**类型：** 学习
**语言：** Python（标准库，无需 LLM）
**前置要求：** Phase 11（LLM 补全 API）
**时间：** 约 45 分钟

## 学习目标

- 解释为什么一个只能生成文本的 LLM 本身无法对真实世界执行操作。
- 画出四步工具调用循环（描述 → 决定 → 执行 → 观察）并指出每步的负责方。
- 将工具描述写成三部分：名称、JSON Schema 输入参数和一个确定性的执行函数。
- 区分纯工具和副作用工具，并说明这种区分对安全性的意义。

## 问题

LLM 输出的是下一个 token 的概率分布。这是全部的输出表面。如果你问一个聊天模型"班加罗尔现在的天气如何"，它可以写出一个听起来合理的句子，但无法接入天气 API。这个句子可能是巧合答对了，或者是三天前的数据。

工具接口正是为了弥合这个差距。宿主机程序——你的 Agent 运行时、Claude Desktop、ChatGPT、Cursor，或自定义脚本——向模型公布一份可调用工具的列表。当模型决定需要某个操作时，它发出一个结构化载荷，点名一个工具及其参数。宿主机解析该载荷，真正运行该工具，然后将结果反馈回去。循环持续，直到模型认为不再需要调用工具。

这份合约的第一个版本于 2023 年 6 月随 OpenAI 的"functions"参数一起发布。Anthropic 在 Claude 2.1 中推出了 `tool_use` 块。Gemini 几个月后添加了 `functionDeclarations`。每个提供商现在都暴露相同的形状：JSON-Schema 类型的工具列表输入，JSON 载荷的工具调用输出。模型上下文协议（Model Context Protocol，2024 年 11 月）将该合约泛化，使一个工具注册表可以为所有模型服务。A2A（2026 年 4 月，v1.0）将同一原语叠加用于 Agent 间委托。

四步循环是其下所有实现的不变基础。Phase 13 的其余内容都是对它的展开。

## 概念

### 步骤一：描述

宿主机用三个字段声明每个工具。

- **名称。** 一个稳定的、机器可读的标识符。用 `get_weather`，而不是"weather thing"。
- **描述。** 一段自然语言简介。"当用户询问特定城市的当前天气条件时使用。不要用于历史数据。"
- **输入 schema。** 一个描述工具参数的 JSON Schema 对象（2020-12 草案）。

模型接收该列表。现代提供商将这些声明序列化为系统提示，使用提供商特定的模板，所以调用方只需处理结构化形式。

### 步骤二：决定

根据用户的消息和可用工具，模型选择三种行为之一。

1. **直接回答。** 用文本。不调用工具。
2. **调用一个或多个工具。** 发出结构化调用对象。在 `parallel_tool_calls: true`（OpenAI 和 Gemini 默认开启，Anthropic 可选开启）下，模型可以在一次对话中发出多个调用。
3. **拒绝。** 严格模式的结构化输出可以产生类型化的 `refusal` 块，而非调用。

工具调用载荷有三个稳定字段：调用 `id`、工具 `name` 和 JSON `arguments` 对象。id 存在是为了让宿主机能将后续结果与特定调用关联起来，这在使用并行调用乱序返回时至关重要。

### 步骤三：执行

宿主机接收调用，验证参数是否匹配声明的 schema，然后运行执行器。参数无效意味着模型幻觉出一个字段或使用了错误的类型——这是弱模型上非常常见的失败模式。生产环境宿主机在参数无效时会做三件事之一：快速失败并将错误暴露给模型，用约束解析器修复 JSON，或将验证错误包含在提示中重试模型。

执行器本身是普通代码。Python、TypeScript、shell 命令、数据库查询。它产生一个结果，通常是字符串，但可以是任何 JSON 值或结构化内容块（MCP 中的文本、图像或资源引用）。结果必须是可序列化的。

### 步骤四：观察

宿主机将工具结果追加到对话中（作为带有匹配 `id` 的 `tool` 角色消息），然后重新调用模型。模型现在在上下文中有了工具输出，可以产生最终答案或请求更多调用。这个过程持续，直到模型停止发出调用，或宿主机达到安全限制（迭代次数）。

### 信任划分

工具有两种风味，对安全性很重要。

- **纯工具。** 只读、确定性、无副作用。`get_weather`、`search_docs`、`get_current_time`。可以安全地 speculative 调用。
- **影响型工具。** 改变状态、花费金钱、接触用户数据。`send_email`、`delete_file`、`execute_trade`。必须加门控。

Meta 2026 年的"二选一原则"（Rule of Two）用于 Agent 安全：一个单轮可以组合最多两个：不可信输入、敏感数据、影响型操作。工具接口正是你执行该规则的地方——通过拒绝调用、要求用户确认，或升级作用域。详见 Phase 13 · 15 的完整安全章节和 Phase 14 · 09 的 Agent 级权限策略。

### 循环所在位置

| 场景 | 谁描述 | 谁决定 | 谁执行 |
|------|--------|--------|--------|
| 单轮函数调用（OpenAI/Anthropic/Gemini） | 应用开发者 | LLM | 应用开发者 |
| MCP | MCP 服务器 | LLM 经由 MCP 客户端 | MCP 服务器 |
| A2A | Agent Card 发布者 | 调用方 Agent | 被调用 Agent |
| Web 浏览器（函数调用 Agent） | 浏览器扩展 / WebMCP | LLM | 浏览器运行时 |

到处都是同样的四步。列名变了，结构没变。

### 为什么不让模型直接输出 JSON？

"让模型回复 JSON"是前函数调用时代的模式。在前沿模型上有 5% 到 15% 的失败率，在更小的模型上失败更多。失败模式包括缺少括号、尾随逗号、幻觉字段和错误类型。然后你需要 JSON 修复、要么重试。

原生函数调用更好，有三个原因。第一，提供商端到端训练模型精确的调用形状，所以有效 JSON 率在严格模式下攀升至 98% 到 99%。第二，调用载荷位于自己的协议槽中，而非自由文本内部——所以工具调用永远不会泄漏到用户可见的回复中。第三，提供商通过约束解码（OpenAI 的严格模式、Anthropic 的 `tool_use`、Gemini 的 `responseSchema`）强制 schema 合规。输出保证通过验证。

### 熔断器

循环在模型停止发出调用或宿主机达到最大轮数时终止。生产环境宿主机将此值设置为 5 到 20 轮之间。超过这个值，几乎肯定是你陷入了模型无法退出的循环。Claude Code 默认 20；OpenAI Assistants 默认 10；Cursor 的 Agent 模式默认 25。

替代方案——无限循环——每半年出现一次："Agent 过夜花费了 400 美元 API 费用"的事后分析。不要在没有界限的情况下发布。

Phase 14 · 12 深入讲错误恢复和自愈；Phase 17 讲生产环境速率限制。

## 使用它

`code/main.py` 无需 LLM 即可运行四步循环。一个假的"决策器"函数通过模式匹配用户消息来模拟模型；执行器、schema 验证器和观察步骤的 harness 是真实的。运行它可以看到完整的请求/响应流程并打印中间状态，然后将假决策器替换为后续课程中任意真实提供商。

要注意的点：

- 工具注册表每个工具保存三个字段：name、description、schema 和执行器引用。
- 验证器是最小化 JSON Schema 子集（类型、required、enum、min/max），只用标准库编写。Phase 13 · 04 发货一个更完整的验证器。
- 循环在五轮时限制迭代次数。生产环境 Agent 正需要这种熔断器。

## 练习

1. 在 `code/main.py` 中添加第四个工具 `get_stock_price(ticker)`。将其描述写为"当用户询问某只股票当前价格时使用。不要用于历史价格或市场总结。"运行 harness 并确认假决策器将提及 ticker 的查询路由到新工具。

2. 破坏 schema 验证器。传递一个缺少 required 字段的调用，确认宿主机在执行前拒绝它。然后传递一个带额外未知字段的调用。决定：宿主机应该拒绝还是忽略？用安全论据为你的选择辩护。

3. 将 harness 中每个工具分类为纯工具或影响型工具。为需要它的注册表条目添加 `consequential: true` 标志，并改变循环，每当选择影响型工具时打印一行"需要与用户确认"。这是每个生产环境宿主机都需要的确认门控的形状。

4. 在纸上画出四步循环，并填写上面提供商列表格中你最喜欢的客户端（Claude Desktop、Cursor、ChatGPT 或自定义技术栈）。与 Phase 13 · 06 中的 MCP 特定变体交叉参考。

5. 从头到尾阅读 OpenAI 的函数调用指南。找出请求中存在但不在这里呈现的四步循环中的一个字段。解释它增加了什么，为什么它是方便的而非必需的。

## 关键术语

| 术语 | 人们通常说 | 实际含义 |
|------|-----------|----------|
| Tool（工具） | "模型可以调用的东西" | 名称 + JSON Schema 类型输入 + 执行器函数的三元组 |
| Function calling（函数调用） | "原生工具使用" | 提供商级 API 支持，用于发出结构化工具调用而非散文 |
| Tool call（工具调用） | "模型的请求操作" | 模型发出的带 `id`、`name`、`arguments` 的 JSON 载荷 |
| Tool result（工具结果） | "工具返回了什么" | 执行器的输出，用匹配 id 的 `tool` 角色消息包装 |
| Parallel tool calls（并行工具调用） | "一次多个调用" | 一个模型轮次中的多个调用对象，按 id 独立排序 |
| Strict mode（严格模式） | "保证 JSON" | 约束解码，强制模型输出通过声明 schema 验证 |
| Pure tool（纯工具） | "只读工具" | 无副作用；可安全重运行 |
| Consequential tool（影响型工具） | "操作工具" | 改变外部状态；需要门控、审计或用户确认 |
| Four-step loop（四步循环） | "工具调用循环" | describe → decide → execute → observe |
| Host（宿主机） | "Agent 运行时" | 持有工具注册表、调用模型并运行执行器的程序 |
