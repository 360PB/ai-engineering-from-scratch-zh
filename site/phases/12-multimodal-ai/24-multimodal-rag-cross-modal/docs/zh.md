# 多模态 RAG 与跨模态检索

> 视觉原生文档 RAG 是一面。生产多模态 RAG 更广——跨文本、图像、音频和视频检索，用于旅行规划（"找一家安静的有自然光的素食早午餐"）、医疗分类（"什么伤势匹配这张照片 + 这些笔记"）、电子商务（"与这张自拍相似的套装，按我的尺码"）和现场服务（"根据引擎声音加零件照片诊断"）等工作流。三项 2025 年调查——Abootorabi 等、Mei 等、Zhao 等——整理了子问题：跨模态检索、检索融合、生成 grounding、多模态评估。本节解读这些调查并设计生产流水线。

**类型：** Build
**语言：** Python（标准库，跨模态检索器 + 融合 + grounded 生成器）
**前置知识：** Phase 12 · 23（ColPali），Phase 11（RAG 基础）
**时间：** 约 180 分钟

## 学习目标

- 设计跨模态检索：文本 → 图像、图像 → 文本、音频 → 视频等。
- 比较三种融合策略：分数融合、注意力融合、MoE 融合。
- 解释生成 grounding：当源是模态混合时，"引用你的来源"是什么样的。
- 说出 2025 年三项规范多模态 RAG 调查及其子问题分类。

## 问题背景

单模态 RAG 是一个已解决的模式：embedding 查询、embedding 块、检索、塞入 LLM。多模态 RAG 需要：

1. 多个检索头（每个模态需要在兼容空间中的 embedding）。
2. 跨模态检索结果融合。
3. 跨模态源的生成 grounding。
4. 覆盖跨模态信号的评估指标。

2025 年的调查都得出了相同的分类。

## 核心概念

### 跨模态检索

给定 A 模态查询检索 B 模态文档。三种模式：

1. 共享 embedding 空间。CLIP 和 CLAP 在共享空间中产生文本 + 图像 / 文本 + 音频 embedding。跨模态直接用余弦相似度。仅限 CLIP 训练的配对。

2. 每模态编码器 + 翻译。文本编码器 + 图像编码器 + 小翻译模块映射空间。Gupta 等人的 Sen2Sen 和其他 2024 年设计。灵活但增加复杂性。

3. VLM 作为编码器。使用 VLM 的隐藏状态作为检索表示。VLM 支持的任何模态都有效。更高质量，更贵。

选择：文本+图像用 CLIP / SigLIP 2；文本+音频用 CLAP；前沿质量跨模态用 VLM 隐藏状态。

### 融合策略

你检索到 10 个结果：5 张图像、3 段文本、2 个音频片段。怎么合并？

分数融合（最便宜）。每个模态有各自的 retriever，各自返回分数。在模态内归一化分数然后求和。简单，通常有效。

注意力融合。拼接所有检索项，让小型注意力网络加权。需要训练。

MoE 融合。门控网络路由到模态特定专家。不同查询类型路由不同——视觉问题对图像加权更高。

生产默认值：分数融合，对查询主要模态略有偏向。如果 A/B 显示在你的领域明确胜出则升级到 MoE。

### 生成 grounding

LLM 应该引用哪个检索项驱动每个声明。对于多模态：

- 文本源：标准引用 `[1]`。
- 图像源：带简短标题的 `[img 3]`。
- 音频：`[audio 2 at 0:34]`。

用 grounding 感知数据训练生成器：训练目标中每个声明标有源索引。推理时模型自然发出引用。

### MuRAG——奠基论文

MuRAG（Chen 等，2022）是首个多模态 RAG。从多模态 KB 中检索图像 + 文本，生成答案。在 VLM 浪潮之前展示可行性。现代系统（REACT、VisRAG、M3DocRAG）建立在其上。

### 生产旅行规划师示例

查询："找一家安静的有自然光的素食早午餐。"

流水线：
1. 分解查询。"安静" → 音频/评论关键词；"素食早午餐" → 菜单项；"自然光" → 图像特征。
2. 每模态检索：
   - 文本检索评论："素食早午餐，安静氛围。"
   - 图像检索餐厅照片："自然光，通风。"
   - 音频检索环境音片段："低分贝，无音乐。"
3. 融合分数。每个餐厅有复合分数。
4. Top-k 餐厅 → 带所有证据的 VLM 生成器 → 带引用的答案。

这远远超出文本 RAG。每个模态添加文本单独错过的信号。

### Agentic 多模态 RAG

多跳：如果第一次检索未返回高置信度答案，LLM 重新表述并检索。Phase 14 的 Agentic RAG 模式在这里适用。例子：

- 检索初始 top-10 → LLM 问"太吵，按 <40 dB 筛选" → 重新检索。
- 检索图像 → LLM 看到一张有菜单 → 检索菜单文本 → 回答。

增加复杂性但处理单发检索无法处理的查询。

### 评估

跨模态评估仍不成熟。常见代理：

- 每模态 Recall@k。
- 融合 top-k 准确率。
- 人类判断的端到端满意度。
- 任务特定（完成的预订、完成的购买）。

没有标准基准覆盖所有模态。大多数论文在特定领域任务上评估。

## 使用方法

`code/main.py`：

- 三个模拟检索器（文本、图像、音频）在共享餐厅语料上操作。
- 分数融合，以可配置权重组合模态分数。
- 生成器 stub，发出带引用的最终答案。
- 简单 agentic 循环，当置信度低时重新表述查询。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| Cross-modal retrieval | "查询一种模态，检索另一种" | 文本查询检索图像；图像查询检索文本；需要共享空间或翻译器 |
| Score fusion | "组合分数" | 每模态检索分数的加权和；最简单的融合 |
| MoE fusion | "模态路由专家" | 门控网络选择每个查询信任哪个模态的分数 |
| Grounded generation | "引用你的来源" | 答案中每个声明标有源索引 |
| MuRAG | "首个多模态 RAG" | 2022 年论文，建立了多模态 RAG 模式 |
| Agentic multi-hop | "重新表述并重试" | 当第一轮置信度低时，LLM 重新查询检索器 |
