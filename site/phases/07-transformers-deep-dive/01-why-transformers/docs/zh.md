# 为什么选择 Transformer — RNN 的问题

> RNN 逐个 token 处理。Transformer 并行处理所有 token。2017 年之后的每一次扩展曲线，都由这个架构赌注所改写。

**类型：** 学习
**语言：** Python
**前置知识：** Phase 3（深度学习核心），Phase 5 · 09（序列到序列），Phase 5 · 10（注意力机制）
**时长：** ~45 分钟

## 问题

2017 年之前，地球上每个最先进序列模型——语言、翻译、语音——都是循环神经网络。LSTM 和 GRU 在 ImageNet 级别的翻译基准上称霸了整整五年。它们是当时唯一的工具。

它们有三个致命弱点。**顺序计算**意味着无法沿时间轴并行化：token `t+1` 需要 token `t` 的隐藏状态。一段 1,024 个 token 的序列意味着在 GPU 上执行 1,024 个串行步骤（GPU 每周期可完成 1,000,000 次浮点运算）。训练墙上时钟时间随序列长度线性增长，而硬件天生为并行设计。

**梯度消失**意味着 50 个 token 之前的信息已经被压缩通过了 50 个非线性函数。门控循环单元（LSTM、GRU）缓解了这种压力，但从未消除它。长距离依赖——"我去年夏天在去京都的飞机上读的那本书……"——经常失败。

**固定宽度隐藏状态**意味着编码器在解码器看到任何东西之前，将整个源序列压缩成单一向量。无论源序列是 5 个 token 还是 500 个，瓶颈形状都是一样的。

2017 年的论文"Attention Is All You Need"提出了一个激进的方案：彻底丢弃循环。令每个位置并行地关注其他每个位置。一次大矩阵乘法完成训练，而非 1,024 个顺序步骤。

结果截至 2026 年主导了所有模态：语言（GPT-5、Claude 4、Llama 4）、视觉（ViT、DINOv2、SAM 3）、音频（Whisper）、生物（AlphaFold 3）、机器人（RT-2）。同一个模块，不同的输入。

## 概念

![RNN 顺序计算 vs Transformer 并行注意力](../assets/rnn-vs-transformer.svg)

**循环作为瓶颈。** RNN 计算 `h_t = f(h_{t-1}, x_t)`。每一步都依赖于上一步。你无法在 `h_4` 之前计算 `h_5`。在拥有 10,000+ 并行核心的现代 GPU 上，这在一段长序列上浪费了 99% 的硅。

**注意力作为广播。** 自注意力同时计算每个 pair `(i, j)` 的 `output_i = sum_j(a_ij * v_j)`。整个 N×N 注意力矩阵在一次批处理矩阵乘法中填满。没有一步依赖于另一步。GPU 喜欢这样。

**加速不是常数。** 是 `O(N)` 串行深度和 `O(1)` 串行深度的差距。在实践中，在 N=512 的匹配硬件上，Transformer 每轮训练快 5–10 倍，序列长度越大差距越大，直到遇到注意力的 `O(N²)` 内存墙（Flash Attention 后来修复了这个问题——见第 12 课）。

**Transformer 的代价。** 注意力内存随 `O(N²)` 增长。2K 上下文还好。128K 上下文需要滑动窗口、RoPE 外推、Flash Attention 分块或线性注意力变体。循环在时间和内存上都是 `O(N)`；Transformer 用时间换内存，然后通过并行性赢回时间。

**归纳偏置的转移。** RNN 假设局部性和近期性。Transformer 不做假设——每个 pair 都是注意力的候选。因此 Transformer 需要更多数据才能训练好，但一旦有了数据就能扩展得更远。Chinchilla（2022）形式化了这一定理：在足够多的 token 条件下，Transformer 总是击败同等参数量的 RNN。

## 构建

这里没有神经网络——我们用数字模拟核心瓶颈，让你实际感受到差距。

### 第一步：测量串行深度

见 `code/main.py`。我们构建两个函数。一个用加法链编码序列（串行，像 RNN）。一个用并行归约编码（广播，像注意力）。相同的数学，不同的依赖图。

```python
def rnn_style(xs):
    h = 0.0
    for x in xs:
        h = 0.9 * h + x   # 无法并行化：h 依赖于上一步的 h
    return h

def attention_style(xs):
    return sum(xs) / len(xs)  # 每个 x 都是独立的
```

我们对最长 100,000 个元素的序列计时。RNN 版本是 O(N) 且是单条 CPU 流水线。即使在纯 Python 中，attention 风格的归约在长度 ≥ 1,000 时就超过了它，因为 Python 的 `sum()` 在 C 中实现，迭代时每步没有解释器开销。

### 第二步：计算理论操作数

两个算法都做 N 次加法。区别是**依赖深度**：下一个操作能开始之前必须顺序发生多少操作。RNN 深度 = N。Attention 深度 = log(N)（树归约）或 1（并行扫描）。深度而非操作数，决定 GPU 时间。

### 第三步：在长序列上做经验扩展

我们打印一个时间表，使 O(N) 差距可见。在 2026 年的 Mac 笔记本电脑上，1,000 以下的序列太快无法测量。100,000 的序列显示一条干净的线性扫描。将其扩展到 16,384-token Transformer 加 12 层 LSTM 等效，你会看到为什么 2016 年的训练墙上时钟是个阻碍。

## 使用

2026 年仍然选择 RNN 的场景：

| 场景 | 选择 |
|------|------|
| 流式推理，一次一个 token，常数内存 | RNN 或状态空间模型（Mamba、RWKV） |
| 非常长的序列（>1M token）注意力内存爆炸 | 线性注意力、Mamba 2、Hyena |
| 没有矩阵乘法加速器的边缘设备 | 深度可分离 RNN 在 FLOPs/瓦特上仍然领先 |
| 其他任何场景（训练、批处理推理、上下文最高 128K） | Transformer |

状态空间模型（SSM）如 Mamba 本质上是带有结构化参数化的 RNN，给它们两全其美的优势：`O(N)` 扫描内存、并行训练（通过选择性扫描）。它们以更好的长上下文扩展恢复 90% 的 Transformer 质量。2026 年大多数前沿实验室训练混合 SSM+Transformer 模型（例如 Jamba、Samba）——循环没有死，它是一个组件。

## 练习

1. **简单。** 取 `code/main.py` 中的 `rnn_style`，将标量隐藏状态替换为长度-64 的隐藏状态向量。重新测量。随着隐藏状态维度增加，串行开销增加了多少？
2. **中等。** 用纯 Python 实现并行前缀和（Hillis-Steele 扫描）。验证它在长度 1024 上产生与串行扫描相同的数值输出。计数深度。
3. **困难。** 将 attention 风格归约移植到 GPU 上的 PyTorch。在序列长度从 64 扫描到 65,536 时为两者计时。绘图并解释曲线形状。

## 关键术语

| 术语 | 大家怎么说 | 实际含义 |
|------|----------|---------|
| 循环（Recurrence） | "RNN 是顺序的" | 计算中步骤 `t` 依赖于步骤 `t-1`，迫使沿时间轴串行执行。 |
| 串行深度 | "图有多深" | 依赖操作的最长链；即使在无限硬件上也限制墙上时钟。 |
| 注意力 | "让 token 互相看" | 加权和 `sum_j a_ij v_j`，其中 `a_ij` 来自位置 i 和 j 之间的相似度得分。 |
| 上下文窗口 | "模型能看到多少" | 一个注意力层可接受为输入的位置数；二次内存代价在此处扩展。 |
| 归纳偏置 | "架构中内置的假设" | 关于数据外观的先验；CNN 假设平移不变性，RNN 假设近期性。 |
| 状态空间模型 | "有代数支撑的 RNN" | 通过结构化状态空间矩阵参数化的循环，用于并行训练。 |
| 二次瓶颈 | "为什么上下文成本如此高" | 注意力内存 = `O(N²)` 随序列长度增长；Flash Attention 隐藏了常数，没有改变扩展性。 |
