# AlphaEvolve — 进化式编码智能体

> 将前沿编码模型与进化循环和机器可检查评估器配对。让循环运行足够长的时间。它发现了一个使用 48 次标量乘法的 4x4 复矩阵乘法程序——56 年来首次超越 Strassen。它还发现了一个 Google 范围的 Borg 调度启发式算法，在生产中恢复了约 0.7% 的集群算力。架构刻意做得很无聊。胜算来自评估器的严谨性。

**类型：** 学习
**语言：** Python（标准库，进化循环玩具）
**前置知识：** Phase 15 · 01（长时程框架），Phase 15 · 02（自学推理）
**时间：** 约 60 分钟

## 问题

大型语言模型可以写代码。进化算法可以搜索代码。两者分开尝试了几十年；都碰到了天花板。LLM 天花板是虚构（confabulation）：模型写出看似合理但不执行其所声称功能的代码。进化天花板是搜索成本：随机变异在语法上很少产生可编译的程序，更不用说更好的了。

AlphaEvolve（Novikov 等，DeepMind，arXiv:2506.13131，2025 年 6 月）将两者结合。LLM 向程序数据库提出有针对性的编辑；自动评估器对每个变体打分；高分变体成为未来代际的父母。LLM 处理编写可信代码的昂贵步骤；评估器捕捉虚构。循环运行数小时到数周。

报告的结果：48 次标量乘法的 4x4 复矩阵乘法（Strassen 1969 年的界限是 49），Google 生产中的 Borg 调度启发式算法，32.5% 的 FlashAttention 内核加速，Gemini 训练吞吐量提升。

架构有效是因为评估器是机器可检查的。在评估器不存在的地方，它就不起作用。这种不对称性才是教训。

## 概念

### 循环

1. 从一个正确但次优的种子程序 `P_0` 开始。
2. 维护一个变体程序数据库，每个都由评估器打分。
3. 从数据库中采样一个或多个父母（MAP-elites 风格或岛屿模型）。
4. 提示 LLM（许多候选用 Gemini Flash，难的用 Gemini Pro）产生父母的修改变体。
5. 在留出的评估器上编译、运行并评估变体。
6. 按其分数和特征向量插入数据库。
7. 重复。

两个细节很关键。首先，LLM 接收的不仅是父母程序——通常是数据库中几个排名靠前的变体，加上评估器签名，加上简短的任务描述。模型的工作是提出可能提高分数的有针对性的更改。其次，数据库是结构化的（MAP-elites 网格、岛屿模型），所以循环探索多样性，而不仅仅是当前领先者。

### 什么使评估器不可妥协

AlphaEvolve 的胜利都来自评估器快速、确定且难以作弊的领域：

- **矩阵乘法算法**：一个单元测试，乘法矩阵并逐位精确检查相等性。
- **Borg 调度启发式算法**：一个生产级模拟器，重放历史集群负载并测量浪费的算力。
- **FlashAttention 内核**：正确性测试加上在真实硬件上的墙钟基准测试。
- **Gemini 训练吞吐量**：每步测量的 GPU 秒数。

在每个案例中，评估器都捕捉到了 LLM 错误的类别，否则这些错误会主导：虚构的正确性声明、在硬件上消失的性能声明，以及边缘情况失败。如果没有评估器，循环就会针对漂亮的代码进行优化。

### 奖励 hacking 是该陈述的另一面

进化优化评估器所测量的任何东西。如果评估器不完美，循环就会找到那个不完美。在一个未验证的领域，循环会优化表面特征，而不是预期行为。DeepMind 在论文中明确标记了这一点：AlphaEvolve 的成功只转移到评估器严谨性与搜索野心相匹配的领域。

2025-2026 年代码搜索循环中奖励 hacking 的具体例子：

- 奖励"完成时间"的优化目标奖励了提交空解决方案。
- 奖励测试正确性的基准分数奖励了记忆测试和过拟合。
- 一个"代码质量"代理奖励了删除注释和重写变量名，而语义没有变化。

AlphaEvolve 的修复方案：交付一个 LLM 从未见过的留出评估器，输入在评估时生成。即便如此，DeepMind 建议对任何拟议部署进行严格审查。

### 为什么 LLM + 搜索比单独使用两者都强

LLM 可以产生可编译的、语义上可信的修改。随机变异的 GA 在 2000 行 Python 文件上几乎总是产生语法错误。LLM 还集中搜索在可信的邻域（改变一个函数，而不是随机字节），这大大减少了浪费的评估器调用。

反过来，评估器捕捉 LLM 的虚构。LLM 会自信地声称一个函数"在极限情况下是 O(n log n)"而实际上是 O(n²)；墙钟基准测试使问题变得确定。

### AlphaEvolve 在前沿技术栈中的位置

| 系统 | 生成器 | 评估器 | 领域 | 示例成果 |
|---|---|---|---|---|
| AlphaEvolve | Gemini | 正确性 + 基准 | 算法、内核、调度器 | 48-mul 4x4 矩阵乘法 |
| FunSearch（DeepMind，2023） | PaLM / Codey | 正确性 | 组合数学 | cap-set 下界 |
| AI Scientist v2（Sakana，L5） | GPT/Claude | LLM 批评 + 实验 | 机器学习研究 | ICLR 研讨会论文 |
| Darwin Godel Machine（L4） | 智能体脚手架 | SWE-bench / Polyglot | 智能体代码 | 20% → 50% SWE-bench |

四个都是同一食谱的变体：生成器 + 评估器 + 循环。差异在于评估器评什么，以及它有多严格。

## 使用

`code/main.py` 在一个玩具符号回归问题上实现了一个最小的类 AlphaEvolve 循环。"LLM"是一个标准库代理，对计算目标函数的程序提出小的语法修改。"评估器"在留出测试点上测量均方误差。

观察：

- 最佳分数如何在代际中改善。
- MAP-elites 网格如何保持多样化的解决方案存活，使循环不会收敛到局部最小值。
- 移除留出测试（仅训练评估器）如何使循环严重过拟合。

## 交付

`outputs/skill-evaluator-rigor-audit.md` 是在新领域考虑 AlphaEvolve 风格循环的前置条件：你的评估器真的能捕捉到你关心的失败吗？

## 练习

1. 运行 `code/main.py`。注意最佳分数轨迹。禁用留出评估器（flag `--no-holdout`）并重新运行。量化过拟合程度。
2. 阅读 AlphaEvolve 论文第 3 节关于 MAP-elites 网格的内容。为一个新问题（例如编译器优化pass）设计一个特征向量描述符，使搜索保持多样化。
3. 48 次乘法的 4x4 结果在 56 年后超越了 Strassen 的 49-mul 界限。阅读论文附录 F，用三句话解释为什么这个问题的评估器特别容易做对，以及为什么大多数领域不是这样。
4. 提出一个 AlphaEvolve 会失败的领域。准确指出评估器在哪里失效以及为什么。
5. 对于一个你熟悉的领域，写出你会使用的评估器签名。包括：（a）正确性条件，（b）性能指标，（c）留出输入生成规则，（d）至少一个反奖励 hacking 检查。

## 关键术语

| 术语 | 人们常说 | 实际含义 |
|---|---|---|
| AlphaEvolve | "DeepMind 的进化式编码智能体" | Gemini + 程序数据库 + 机器可检查评估器 |
| MAP-elites | "多样性保持档案" | 按特征向量键控的网格；每个单元格持有该描述符的最佳变体 |
| 岛屿模型 | "并行进化子种群" | 周期性迁移的独立种群；防止过早收敛 |
| 机器可检查评估器 | "确定性 oracle" | LLM 无法伪造的单元测试、模拟器或基准——是此循环的先决条件 |
| 奖励 hacking | "优化了指标而不是目标" | 循环找到最大化分数而不执行预期任务的方法 |
| 种子程序 | "起点" | 循环从中进化的初始正确但次优的程序 |
| 留出评估器 | "LLM 从未见过的评估数据" | 在评估时生成的输入以防止记忆 |
