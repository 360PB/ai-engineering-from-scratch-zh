# 长期后台智能体：持久执行

> 生产级长时域智能体不是运行在 `while True` 中的。每个 LLM 调用都成为一个带检查点、重试和回放的活动。Temporal 的 OpenAI Agents SDK 集成于 2026 年 3 月 GA。Claude Code 例程（Anthropic）运行调度的 Claude Code 调用，无需持久本地进程。会话在人类输入时暂停，存活部署，并在按 `thread_id` 键控的最新检查点恢复。新的人体工程学背后是一个旧模式——工作流编排——只有一个新输入：LLM 调用作为必须确定性回放的非确定性活动。

**类型：** Learn
**语言：** Python（标准库，最小持久执行状态机）
**前置要求：** Phase 15 · 10（权限模式），Phase 15 · 01（长时域智能体）
**时间：** 约 60 分钟

## 问题

考虑一个运行四小时的智能体。它调用三个工具，两次提示用户，四十次 LLM 调用。运行到一半，所在主机重启。会发生什么？

- 在天真的 `while True` 循环中：一切丢失。运行从头重新开始。三个工具调用（带有真实副作用）再次执行。用户再次被提示已批准的事项。四十次 LLM 调用被重新计费。
- 使用持久执行：运行从最近检查点恢复。已经完成的活动不会重新执行；它们的結果从持久日志回放。用户不会重新批准已批准的事项。已经进行的 LLM 调用不会被重新计费。

这与工作流引擎已经交付十年的模式相同（Temporal、Cadence、Uber 的 Cherami）。新的是 LLM 调用现在是一种活动——非确定性、昂贵、有副作用——它们很好地符合这个模式。

本课的主题：长时域可靠性会衰减（METR 观察到"35 分钟退化"——成功率大约与时域成二次方下降）。持久执行使运行能够比可靠性曲线支持的更长，这是一种新的安全失败方式——如果设计正确的话是安全的，如果设计错误的话是不安全的。

## 概念

### 活动、工作流和回放

- **工作流**：确定性编排代码。定义活动序列、分支、等待。必须是确定性的，以便可以从事件日志回放而不会产生意外的偏离。
- **活动**：非确定性的、可能失败的工作单元。LLM 调用、工具调用、文件写入、HTTP 请求。每个活动都带有其输入和（一旦完成）其输出的日志。
- **事件日志**：持久备份存储。每个活动开始、完成、失败、重试，以及每个工作流决策都被记录。
- **回放**：恢复时，工作流代码从头重新运行；每个已经完成的活动返回其日志结果而不重新执行。只有尚未完成的活动才实际运行。

这与 React 对虚拟 DOM 重新渲染，或 Git 从提交重建工作树是相同形状。编排器中的确定性使持久性变得廉价。

### 为什么 LLM 调用符合该模式

LLM 调用是：
- 非确定性的（temperature > 0；即使 temperature 0 也因模型版本而漂移）。
- 昂贵的（金钱和延迟）。
- 可能失败的（速率限制、超时）。
- 有副作用的（如果调用工具）。

这正是活动的配置。将每个 LLM 调用包装为活动，给你带指数退避的重试、跨重启的检查点，以及用于调试的可回放跟踪。

### 按 `thread_id` 键控的检查点

LangGraph、微软智能体框架、Cloudflare Durable Objects 和 Claude Code 例程都收敛到相同的 API 形状：`thread_id`（或等效项）标识会话；每个状态转换持久化到后端（PostgreSQL 默认，SQLite 用于开发，Redis 用于缓存）；恢复读取最新检查点。

后端选择很重要：

- **PostgreSQL**：持久、可查询、存活部署。LangGraph 默认。
- **SQLite**：仅本地开发；跨主机丢失数据。
- **Redis**：快速但临时，除非配置了 AOF/快照。
- **Cloudflare Durable Objects**：透明分布式；按唯一键范围；存活数小时到数周。

### 人类输入作为一等状态

先提议后提交（第 15 课）需要持久的"等待人类"状态。工作流暂停，外部队列持有待定请求，批准从确切的那一点恢复。没有持久性，这是尽力而为；有了它，隔夜批准到达，工作流在早上恢复。

### 35 分钟退化

METR 观察到每个测量的智能体类别在连续运行约 35 分钟后可靠性都会下降。将任务持续时间加倍可将失败率提高约四倍。持久执行不能修复这个；它允许你运行比可靠性曲线支持的更长。安全模式是将持久性与重新进入时需要新 HITL 的检查点相结合，并与预算 kill 开关（第 13 课）相结合，无论墙上时钟时间如何，都限制总计算量。

### 持久执行是错误答案时

- 短于几分钟且无人输入的运行。开销 > 收益。
- 严格只读信息检索。
- 正确性需要在单个上下文窗口内端到端完成的任务（一些推理任务；一些一次性生成）。

## 使用它

`code/main.py` 用标准库 Python 实现了一个最小持久执行引擎。它支持：

- `@activity` 装饰器，将输入和输出记录到 JSON 事件日志。
- 一个编排活动的工作流函数。
- 一个 `run_or_replay(workflow, event_log)` 函数，回放已完成的活动而不重新执行。

驱动模拟一个三活动工作流，中途崩溃，并显示 (a) 天真重试重新执行一切 vs (b) 回放只运行缺失的活动。

## 交付它

`outputs/skill-durable-execution-review.md` 审查拟议长期智能体部署的正确持久执行形态：活动、确定性、检查点后端、人类输入状态和 HITL 恢复策略。

## 练习

1. 运行 `code/main.py`。观察天真重试和回放之间的活动执行计数差异。更改崩溃点并显示回放计数相应变化。

2. 将玩具引擎转换为显式使用 `thread_id`。模拟两个共享引擎的并发会话并确认它们的事件日志不会冲突。

3. 取玩具引擎中的一个活动。引入非确定性（工作流决策中的墙上时钟时间戳）。演示回放时的偏离。解释真实引擎如何处理这个问题（副作用注册、`Workflow.now()` API）。

4. 阅读 LangChain"生产深度智能体背后的运行时"帖子。列出运行时持久化的每个状态，并命名每个状态覆盖的失败模式。

5. 为 6 小时自主编码任务设计检查点策略。你在哪里检查点？恢复崩溃后是什么样？什么需要新 HITL？

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|---|---|---|
| 工作流 | "智能体的脚本" | 确定性编排代码；从事件日志可回放 |
| 活动 | "一步" | 非确定性单元（LLM 调用、工具调用）；前后记录 |
| 事件日志 | "备份存储" | 每个状态转换的持久记录 |
| 回放 | "恢复" | 重新运行工作流；已完成的活动返回日志结果而不重新执行 |
| 检查点 | "保存点" | 按 thread_id 持久化的状态；恢复时 latest-wins |
| thread_id | "会话键" | 作用域持久状态的标识符 |
| 35 分钟退化 | "可靠性衰减" | METR：成功率大约与时域成二次方下降 |
| 非确定性 | "回放时漂移" | 墙上时钟、随机、LLM 输出；必须注册为副作用 |
