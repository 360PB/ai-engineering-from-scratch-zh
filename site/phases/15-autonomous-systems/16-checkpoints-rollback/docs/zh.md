# 检查点与回滚

> 每个图状态转换都持久化。当 worker 崩溃时，租约过期，另一个 worker 在最新检查点接续。Cloudflare Durable Objects 跨小时或周保持状态。先提议后提交（第15课）为每个动作定义回滚计划。动作后验证关闭循环。欧盟 AI 法案第14条使有效人工监督对高风险系统成为强制——实践中这意味着检查点必须可查询，回滚必须排练，审计追踪必须能扛住部署。

**类型：** Learn
**语言：** Python（stdlib，检查点和回滚状态机）
**前置知识：** Phase 15 · 12（持久化执行），Phase 15 · 15（先提议后提交）
**时长：** 约60分钟

## 问题

持久化执行（第12课）使崩溃的 Agent 可恢复。先提议后提交（第15课）使批准的动作可审计。本课将两者连接：当批准的动作执行一半、崩溃并恢复时会发生什么？回滚何时跑，针对什么状态？

真实系统以不同方式接线：

- **LangGraph** 检查点每个图状态转换到 PostgreSQL。Worker 崩溃时，租约释放，另一个 worker 从最新检查点恢复。
- **Cloudflare Durable Objects** 跨小时或周保持每键状态。
- **Microsoft Agent Framework** 在工作流 API 中暴露 `Checkpoint` 原语；幂等性加 replay 覆盖重试。

每种情况真正有效的组合：幂等性 key（防止重复执行）+ 前置条件检查（状态仍是我们批准的）+ 动作后验证（副作用实际发生了）+ 验证失败时回滚。

## 核心概念

### 每个转换都持久化

图状态转换是从一个命名状态到另一个的任意步骤。天真实现只在特定提交点持久化；生产实现在每个转换都持久化。成本（几个额外写入）相对可靠性收益（replay 落在任何地方，租约恢复精确）很小。

### 租约恢复

Worker 崩溃时，工作流不丢失；租约（这个 worker 正在执行这个 run 的短期声明）简单过期。另一个 worker 领取最新检查点并恢复。租约机制让生产系统能度过滚动部署而不丢进行中的工作。

### 幂等性加前置条件

仅幂等性不够。考虑：工作流被批准"当余额 > $1000 时从 A 转 $100 到 B"。工作流提交、崩溃于执行中途、恢复。如果只检查幂等性 key，执行恢复，转移运行一次（正确）。但考虑崩溃和恢复之间 A 的余额通过另一个工作流已降至 $500。幂等性检查仍通过；前置条件不通过。无前置条件检查，我们发货了透支。

每个重要动作都需要两者：

- **幂等性 key**：防止重复执行。
- **前置条件检查**：确认状态仍与我们批准的一致。

### 动作后验证

"工具返回 200"不是验证。真实验证重新读目标状态并确认副作用实际发生了。模式：

- 数据库更新：`UPDATE ... RETURNING *` 然后断言返回行匹配预期状态。
- 邮件发送：提交后在已发送文件夹中检查消息 ID。
- 文件写入：读回文件并哈希。
- API 调用：对目标资源跟进 `GET`。

如果验证失败，工作流处于已知坏状态。回滚介入。

### 回滚计划

先提议后提交（第15课）中每个重要动作都带回滚计划。类型：

- **带内回滚**：直接反转副作用（`INSERT` 后 `DELETE`，发送后发更正邮件）。
- **补偿事务**：中和原始动作的新动作（标准 SAGA 模式）。
- **带外回滚：告警人工，暂停工作流，留坏状态待调查。

无操作回滚（"我们无法撤销这个"）必须在提议中命名。无回滚的动作在提交时需要更强 HITL。

### 欧盟 AI 法案第14条操作解读

第14条要求高风险系统的"有效人工监督"。操作上，实现者解读为：

- 检查点可被审计员查询。
- 回滚已排练（至少端到端测试一次）。
- 审计追踪扛住部署（检查点后端非临时的）。
- 失败验证告警，而非静默记录。

## 练习

1. 跑 `code/main.py`。验证四个场景。对于崩溃-提交场景，确认动作跨重试恰好执行一次。
2. 修改"先标记完成再执行"模式，使状态写入在动作后触发。重跑崩溃场景。测量多少重复动作触发。
3. 为特定生产动作（如"发 Slack 频道"）设计回滚计划。分类为带内、补偿或带外。证明选择。
4. 取一个你了解的工作流。识别每个状态转换。标记每个的持久化需求（持久化 / 不持久化）。数出你当前未持久化的数量。
5. 排练回滚测试：设计端到端测试，跑真实工作流，崩溃它，确认回滚路径触发。测试断言什么？

## 关键术语

| 术语 | 行话 | 实际含义 |
|------|---|---|
| Checkpoint | "保存点" | 每个图状态转换持久化到持久存储 |
| Lease | "Worker 声明 | Worker 正在执行 run 的短期声明；崩溃时过期 |
| Precondition | "状态门控" | 状态仍与批准动作一致的断言 |
| Post-action verify | "重新读检查" | 确认副作用实际在目标系统中发生了 |
| In-band rollback | "直接撤销" | 用逆操作反转副作用 |
| Compensating transaction | "SAGA 撤销 | 中和原始动作的新动作 |
| Mark-as-done-first | "状态写入顺序 | 在提交返回前持久化为完成状态 |
| Article 14 | "欧盟 AI 法案人工监督 | 操作上：可查询检查点、排练回滚、可审计追踪 |
