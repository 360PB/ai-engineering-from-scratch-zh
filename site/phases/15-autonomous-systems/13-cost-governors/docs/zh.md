# 操作预算、迭代上限和成本调节器

> 一个中型电商智能体的月度 LLM 成本在团队启用"订单追踪"技能后从 $1,200 跳到 $4,800。这不是定价 bug。这是一个找到新循环并在其中持续消费的智能体。微软的智能体治理工具包（2026 年 4 月 2 日）编纂了对这类失败的防御：每请求 `max_tokens`、每任务 token 和美元预算、每天/每月上限、迭代上限、分层模型路由、提示缓存、上下文窗口化、对昂贵操作的人工介入检查点、对预算违规的 kill 开关。Anthropic 的 Claude Code Agent SDK 以不同名称提供相同的原语。金融速率限制——例如 10 分钟内超过 $50 则切断访问——比月度上限更快地捕获循环。

**类型：** Learn
**语言：** Python（标准库，分层成本调节器模拟器）
**前置要求：** Phase 15 · 10（权限模式），Phase 15 · 12（持久执行）
**时间：** 约 60 分钟

## 问题

自主智能体在每轮都花费真金白银。聊天机器人的糟糕输出是一条糟糕的回复；智能体的糟糕循环是一笔账单。行业记录的失败模式术语是"钱包拒绝"——智能体持续推理、持续调用工具、持续计费，没有什么阻止它，因为没有设计什么来阻止它。

修复不是单一数字。它是不同时间尺度和粒度的一系列限制：每请求、每任务、每小时、每天、每月。设计良好的栈在几分钟内捕获失控循环，在几小时内捕获缓慢泄漏，在一天内捕获糟糕发布。当智能体是长时域和自主的时候，相同的栈将预算保持在合理范围内。

这是一堂工程课：数学很简单，团队失败的纪律在哪里。下面的限制列表都已在微软智能体治理工具包或 Anthropic Claude Code Agent SDK 文档中命名。

## 概念

### 成本调节器栈

1. **`max_tokens` 每请求。** 简单。防止任何一次调用发出无界补全。
2. **每任务 token 预算。** 跨整个运行，不要超过 N 个 token。在上限处硬停止。
3. **每任务美元预算。** 与 token 相同但以货币计。Claude Code 中的 `max_budget_usd`。
4. **每个工具调用上限。** 不超过 N 次 `WebFetch` 调用、N 次 `shell_exec` 调用等。
5. **迭代上限（`max_turns`）。** 总智能体循环迭代次数；防止无限推理循环。
6. **每分钟/每小时/每天/每月上限。** 滚动窗口。在不同时间尺度上捕获泄漏。
7. **金融速率限制。** 例如，"如果 10 分钟内支出超过 $50，切断访问"。在月度上限触发前捕获基于循环的燃烧。
8. **分层模型路由。** 默认为较小的模型；仅当分类器判断任务需要时才升级到较大的模型。
9. **提示缓存。** 系统提示和稳定上下文存储在提供者缓存中；重新发送的 token 成本接近零。
10. **上下文窗口化。** 压缩/总结以将活动上下文保持在阈值以下；直接减少 token 成本。
11. **对昂贵操作的人工介入检查点。** 在已知昂贵的操作（长工具调用、大量下载、昂贵的模型升级）之前，需要人类点击。
12. **预算违规的 kill 开关。** 任何上限触发时中止会话。记录上限；需要单独的重新启用路径。

### 为什么是栈，而不是单一上限

单一月度上限只在钱包耗尽后才捕获失控的智能体。单一每请求上限在会话级别捕获不了什么。不同的失败模式需要不同的时间尺度：

- **失控循环**（智能体卡在 5 秒重试中）：被速率限制捕获。
- **缓慢泄漏**（智能体做约 2 倍于预期每任务的工作）：被每日上限捕获。
- **糟糕发布**（新版本使用 5 倍 token）：被每周/每月上限捕获。
- **合法激增**（真实需求，不是 bug）：被每小时/每天上限捕获，并带有清晰的日志。

### Claude Code 的预算面

Claude Code Agent SDK 暴露（公开文档）：

- `max_turns` — 迭代上限。
- `max_budget_usd` — 美元上限；违规时中止会话。
- `allowed_tools` / `disallowed_tools` — 工具允许列表和拒绝列表。
- 工具使用前的钩子点，用于自定义成本核算。

与权限模式阶梯结合（第 10 课）。没有 `max_budget_usd` 的 `autoMode` 会话是不受管治的自主性。Anthropic 明确将自动模式框架为需要预算控制；分类器与成本无关。

### EU AI 法案，OWASP 智能体十大

微软的智能体治理工具包涵盖 OWASP 智能体十大和 EU AI 法案第 14 条（人类监督）要求。在欧盟生产，登录和上限执行不是可选的。

### 观察到的 $1,200 → $4,800 案例

微软文档中的一个真实案例：一个电商智能体，其月度成本在添加新工具后增加了两倍。该工具允许智能体在每个会话中轮询订单状态。没有循环检测。没有每个工具的上限。没有周环比增长的警报。修复是每个工具上限加每日增长警报。这是一个模板：每个新工具面都是一个新潜在循环；每个新工具都需要自己的上限和自己的警报。

## 使用它

`code/main.py` 模拟有和没有分层成本调节器栈的智能体运行。模拟智能体在某些轮次后漂移到轮询循环；分层栈在速率窗口内捕获它，而单一月度上限会在数天后才触发。

## 交付它

`outputs/skill-agent-budget-audit.md` 审计拟议智能体部署的成本调节器栈并标记缺失的层。

## 练习

1. 运行 `code/main.py`。确认速率限制在轮询循环轨迹上比迭代上限先触发。现在禁用速率限制并测量智能体"花费"多少，直到迭代上限捕获它。

2. 为浏览器智能体设计每工具上限集（第 11 课）。哪个工具需要最严格的上限？哪个工具可以无限制运行而没有风险？

3. 阅读微软智能体治理工具包文档。列出工具包命名的每种上限类型。将每个映射到一种失败模式（失控循环、缓慢泄漏、糟糕发布、激增）。

4. 为真实任务（例如"整理仓库中的 50 个问题"）的隔夜无人值守运行定价。将 `max_budget_usd` 设置为你的点估计的 2 倍。为 2 倍提供理由。

5. Claude Code 的 `max_budget_usd` 在会话总成本上触发。设计一个你将在外部强制执行的补充速率限制。什么触发切断，重新启用是什么样的？

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|---|---|---|
| 钱包拒绝 | "失控账单" | 智能体循环产生支出，没有上限来阻止它 |
| max_tokens | "每请求上限" | 单一补全大小的上限 |
| max_turns | "迭代上限" | 会话中智能体循环迭代的上限 |
| max_budget_usd | "美元 kill 开关" | 会话成本上限；违规时中止 |
| 速率限制 | "速率上限" | 短窗口内的支出限制（例如 $50/10 分钟） |
| 分层路由 | "小模型优先" | 便宜模型默认；仅当分类器必要时升级 |
| 提示缓存 | "缓存的系统提示" | 提供者侧缓存将重新发送的 token 成本降低到接近零 |
| HITL 检查点 | "人类批准门控" | 昂贵操作之前需要人类点击 |
