# 流式语音到语音 — Moshi、Hibiki 与全双工对话

> 2024-2026 年重新定义了语音 AI。Moshi 用单一模型同时听和说，延迟 200ms。Hibiki 逐块做语音到语音翻译。两者都用统一的全双工架构放弃了 ASR → LLM → TTS 流水线。

**类型：** Learn
**语言：** Python
**前置知识：** Phase 6 · 13（神经音频编解码），Phase 6 · 11（实时音频），Phase 7 · 05（完整 Transformer）
**时长：** 约75分钟

## 问题

从第11-12课构建的每个语音 Agent 有约 300-500ms 的根本延迟下限：VAD 触发、STT 处理、LLM 推理、TTS 生成。每个阶段有其自己的最低延迟。你可以调优和并行化，但流水线形态封死了你。

Moshi（Kyutai，2024-2026）问了不同的问题：如果没有流水线呢？如果一个模型直接输入音频输出音频，持续，用文本作为中间"内心独白"而非必需阶段呢？

答案是**全双工语音到语音**。理论延迟 160ms（80ms Mimi 帧 + 80ms 声学延迟）。实际延迟单张 L4 GPU 上 200ms。这是一流流水线语音 Agent 的一半。

## 核心概念

### Moshi 架构

**输入。** 两个 Mimi 编解码流，都在 12.5Hz × 8 个码本：

- 流 1：用户音频（Mimi 编码，持续到达）
- 流 2：Moshi 自己的音频（Moshi 生成的）

**Transformer。** 7B 参数 Temporal Transformer 处理两个流和文本"内心独白"流。每 80ms 步：

1. 消费最新用户 Mimi token（8个码本）。
2. 消费最新 Moshi Mimi token（8个码本，如生成的）。
3. 生成下一个 Moshi 文本 token（内心独白）。
4. 生成下一个 Moshi Mimi token（通过小型 Depth Transformer）。

### 为什么内心独白文本有帮助

没有显式文本，模型必须在声学流中隐式建模语言。Moshi 的洞察：强制它同时发出文本 token。文本流本质上是 Moshi 正在说的转录。这改善了语义连贯性，使换出语言模型头更容易，并免费给你转录。

### Hibiki：流式语音到语音翻译

相同架构，在翻译对上训练。源语言音频输入，目标语言音频输出，持续进行。

## 关键术语

| 术语 | 实际含义 |
|------|---|
| 全双工 | 同时听和说，无流水线间隙 |
| 内心独白 | 模型同时输出的文本 token，作为连贯性的辅助 |
| Mimi 编解码 | Kyutai 的神经音频编解码，80ms 帧 |
| 流式翻译 | 实时将语音翻译为另一语言，无 ASR → LLM → TTS |
| Depth Transformer | 预测帧内 8 个码本的顺序依赖 |
