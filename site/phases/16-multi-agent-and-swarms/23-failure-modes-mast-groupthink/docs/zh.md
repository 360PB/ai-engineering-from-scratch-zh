# 失败模式 — MAST、群体思维、单一文化、级联故障

> 2026 年的参考分类法是 **MAST**（Cemri 等，NeurIPS 2025，arXiv:2503.13657），来自 1642 条执行轨迹横跨 7 个最先进的开源多 Agent 系统，失败率 **41–86.7%**。三个根本类别：**规范问题**（41.77%）——角色模糊、任务定义不清；**协调失败**（36.94%）——通信中断、状态不同步；**验证缺口**（21.30%）——缺少校验、缺乏质量检查。**群体思维**（Groupthink）家族（arXiv:2508.05687）补充：单一文化崩溃（相同基模型 → 相关性故障）、conformity 偏差（Agent 互相强化错误）、心智理论缺陷、混合动机动态、级联可靠性故障。级联示例：支付失败触发订单重试，订单重试触发库存查询，库存服务 10 秒内负载暴增 10 倍——需要断路器。记忆污染：某个 Agent 的幻觉进入共享内存，下游 Agent 当作事实；准确率缓慢衰减，根因诊断困难。**STRATUS**（NeurIPS 2025）报告专用检测/诊断/验证 Agent 可将缓解成功率提升 1.5 倍。本节将失败模式视为一等工程目标。

## 问题背景

多 Agent 系统在真实任务上的失败率是 41-86.7%（Cemri 等 2025 年对 7 个开源 MAS 的测量）。这不是"多加几个 Agent"就能 debug 的问题。失败有结构性成因。MAST 分类法给了你类别。本节将每个类别映射到具体的检测、诊断和缓解模式，让那些数字不再是冷冰冰的统计。

2026 年的生产实践是将失败模式作为设计输入。你的架构在能指出每个 MAST 类别并说出你部署了什么缓解措施之前，不能算"够好"。

## 概念

### MAST 类别

**规范问题（占故障的 41.77%）。** Agent 的任务定义不够严密。例如：

- 角色模糊：两个 Agent 都认为自己是审核者。
- 任务描述不足："总结一下"，而用户想要的是一个特定角度的总结。
- 成功标准不明确：Agent 无法判断自己是否成功了。

缓解措施：
- 写出明确的角色契约。每个 Agent 的提示词中说明它**做什么**，也说明它**不做什么**。
- 每个任务有验收测试。Agent 开始前定义"完成的样子是 X"。
- 飞前规范检查：派一个独立 Agent 在分发任务前审查任务定义。

**协调失败（36.94%）。** 通信或状态中断。

例如：
- 两个 Agent 同时更新共享状态而没有同步。
- Agent 之间的消息丢失（队列故障、超时）。
- 状态漂移：Agent A 认为任务已完成；Agent B 还在执行。

缓解措施：
- 带版本控制的共享状态 + 乐观并发。
- 关键消息的显式确认（重试直到确认）。
- 周期性状态同步检查点；尽早检测漂移。

**验证缺口（21.30%）。** 输出没有独立检查。

例如：
- 某 Agent 声称成功，没人验证。
- 一连串 Agent 每个都信任前一个的输出。
- 组合行为的测试覆盖缺失。

缓解措施：
- 独立验证 Agent（见第 13 章）。只读、独立数据源访问。
- 显式交接合同："A 的输出必须通过检查者 C 的验证，B 才能开始"。
- 结果日志用于事后分析。

### 群体思维家族（arXiv:2508.05687）

当 Agent 同质化或互相模仿时，五种相关失败：

**单一文化崩溃。** 相同的基模型或训练数据 → 相关性错误。当三个 Agent 共享同一个 LLM 时，它们也共享它的幻觉。

**Conformity 偏差。** Agent 向最响亮或最自信的同行靠拢，即使对方是错的。

**心智理论缺陷。** Agent 无法建立对彼此信念的模型；协调崩溃（第 18 章）。

**混合动机动态。** 利益部分对齐的 Agent 向折中妥协漂移，而折中谁都不满意。

**级联可靠性故障。** 一个组件的错误模式触发依赖组件的错误模式。

### 级联示例 — 重试风暴

一个经典的 2026 年事故模式：

```
支付服务 10% 请求失败
   ↓
订单 Agent 重试支付（指数退避但实现简陋）
   ↓
每次重试都是一个新的订单-库存查询
   ↓
库存服务看到 2 倍正常负载
   ↓
库存服务开始超时
   ↓
每个订单都在重试库存查询
   ↓
库存服务看到 10 倍正常负载
   ↓
集群宕机
```

解决方法很经典：**断路器**。当下游错误率超过阈值时，用缓存或默认结果短路。加上每个请求的重试预算上限。

断路器是为数不多的从分布式系统直接借鉴过来、不用修改就能用于多 Agent 系统的失败缓解措施之一。

### 记忆污染（回访第 13 章）

第 13 章已覆盖：一个 Agent 的幻觉变成共享内存中的事实；下游 Agent 在这个被污染的事实上推理。在 MAST 术语中，这是共享内存层的验证缺口。

症状是准确率缓慢下降。系统不会崩溃；是缓慢漂移，难以根因定位。

缓解：只追加日志、溯源、不可写入的验证器。第 13 章已覆盖。

### STRATUS — 专用的失败检测 Agent

STRATUS（NeurIPS 2025）报告当你部署以下三种 Agent 时，缓解成功率提升 1.5 倍：

- **检测 Agent。** 监控症状模式（高分歧、重试峰值、准确率漂移）。
- **诊断 Agent。** 给定症状，从 MAST 分类推断可能的根因。
- **验证 Agent。** 缓解措施应用后，检查症状是否消失。

这是 SRE 风格的事故响应，应用于 Agent 系统。三个角色都可以是带有专门提示词的 LLM Agent。

### 失败模式审计

2026 年的最佳实践是每年（或每重大版本）做一次失败模式审计：

1. **采集轨迹样本。** 收集约 1000 条真实执行轨迹。
2. **分类。** 对每条轨迹的失败，映射到 MAST + 群体思维类别。
3. **计算各类别失败率。** 你的系统哪类故障最多？
4. **排名缓解措施。** 哪个修复能消除最多失败？
5. **选 2-3 个缓解措施。** 实施；下个季度重新审计。

纪律比具体选择更重要。没有审计，失败会混入噪声，永远不会被系统性解决。

### 系统静默失败时

最危险的失败类别是静默正确性失败。失败大声的系统（崩溃、异常、告警）可以被监控。产生可信但错误输出的系统无法通过异常日志检测。这正是验证缺口虽然按数量只占 21.30%，但每次失败代价最贵的原因。

需要在以下方面投入：
- 抽样人工审查。
- 金标准回归测试集。
- 重要输出的跨 Agent 交叉检查。

### 失败 vs 慢失败

有些失败是立即的；有些是缓慢的。立即失败（超时、schema 不匹配、认证错误）检测成本低。慢失败（记忆污染、单一文化漂移、角色模糊）检测和预防成本高。

2026 年的工程行动：监控慢失败的代理指标，以便在变成可见错误之前捕获漂移。一致率、重试率、输出长度分布、相邻 Agent 版本间的编辑距离，都是有用的代理指标。

## 产出

生产中的失败模式纪律：

- **每季度做 MAST 审计。** 不是每年。类别会随系统增长而变化。
- **处处加断路器。** 每个对下游服务的出站调用都要有。默认阈值 5-10% 错误率。
- **金标准数据集。** 小规模、高质量、人工审查。每周用它们做回归测试。
- **STRATUS 三件套。** 检测 + 诊断 + 验证 Agent 监控生产。先只部署检测 Agent；症状嘈杂时再加入诊断。
- **失败预算。** 明确各类别失败率的 SLO。超过预算触发停止发版讨论。

3. 阅读 Cemri 等（arXiv:2503.13657）。选取他们的 7 个 MAS 系统之一，映射其前 3 大失败类别。这些与 MAST 预测相比如何？

5. 为你熟悉的多 Agent 系统设计 STRATUS 风格的检测-诊断-验证三件套。检测 Agent 监控哪些症状？诊断 Agent 推荐哪些缓解措施？验证 Agent 如何确认它们有效？
