# 并行 / 蜂群 / 网络化架构

> 与主管对比：没有中央决定者。智能体读取共享事件总线，异步获取工作，写回结果。LangGraph 明确支持"蜂群架构"用于去中心化、动态环境。Matrix（arXiv:2511.21686）将控制流和数据流都表示为通过分布式队列传递的序列化消息，消除编排器瓶颈。权衡是明确的：可确定性及可追溯性换取可扩展性。蜂群适合具有许多独立子问题的任务；不适合需要单一连贯计划的任务。

**类型：** 学习 + 构建
**语言：** Python（标准库、`threading`、`queue`）
**前置知识：** Phase 16 · 05（主管模式），Phase 16 · 04（原语模型）
**时间：** 约75分钟

## 问题

主管扩展到几个工作器。数百个呢？主管本身就成为瓶颈：关于谁做什么的每个决定都经过一个智能体。一个慢计划步骤 stall 整个系统。

蜂群架构翻转了设计。不是中央规划器dispatch 工作，工作器从共享队列中拉取工作。"协调"烘焙进事件总线语义。没有编排器；系统扩展直到队列扩展。

## 概念

### 形状

```
                ┌──── 共享队列 ────┐
                │                      │
       ┌────────┼────────┐  ◄──────┬───┘
       ▼        ▼        ▼         │
     工作器  工作器  工作器   工作器
      A       B       C        D
       │        │        │         │
       └────────┴────────┴─────────┘
                 │
                 ▼
            结果池
```

没有编排器。每个工作器重复：拉取任务、处理、写结果（并可选 enqueue follow-up）。

### 何时蜂群适合

- **许多独立任务。** 抓取、转换、分类。任务不互相依赖。
- **可变时长工作。** 如果一些任务需要 100ms 其他需要 10s，蜂群自动平衡负载——快工作器拉取下一工作。主管必须预估时长。
- **吞吐量胜于确定性。** 你关心总完成时间，不是严格顺序。

### 何时蜂群失败

- **顺序工作流。** 如果步骤 3 需要步骤 2 的输出，蜂群冒步骤 3 在步骤 2 完成前触发的风险。
- **全局计划任务。** 复杂研究问题受益于规划器。蜂群研究产生独立事实，不是连贯报告。
- **调试。** 没有中央日志和异步工作，再现 bug 代价高昂。

### Matrix（arXiv:2511.21686）

Matrix 是 2026 年论文，将蜂群推到其自然结论：控制流和数据流都是分布式队列上的序列化消息。没有中央协调器。容错来自消息持久性。可扩展性是消息代理的问题，不是系统的问题。

贡献：编程模型，多智能体协调是"此智能体订阅什么消息主题？"而不是"编排器选择下一个智能体？"这使系统看起来像 pub/sub 事件网格。

### LangGraph 的蜂群架构

LangGraph 2025 文档明确将"蜂群架构"描述为多智能体模式之一：智能体是节点，但边形成有环的有向图，任何节点可从池中激活。工作器通过条件从可用工作中选取，不是通过主管分配。

### 失败模式：饥饿和热点

如果所有工作器拉取最快可用任务，长运行任务永远不会被选取直到它们是唯一剩余的。经典队列饥饿。

缓解：
- 优先级队列，带显式老化（等待时间增加优先级）。
- 工作器专业化：一些工作器只取"长"任务。
- 背压：限制多少快任务进入队列。

### 内容相关路由链接

蜂群自然与内容相关路由（Lesson 22）配对。不是泛型队列，而是每个消息类型一个队列。专家工作器只订阅其类型。这构成了可扩展到数千智能体的消息总线架构的基础。

## 构建

`code/main.py` 实现 4 个工作线程从共享 `queue.Queue` 拉取的蜂群。任务有可变时长（一些快，一些慢）。演示对比：

- **顺序基线：** 一个工作器串行处理所有任务。
- **固定分配：** 每个任务预分配给特定工作器（主管风格）。
- **蜂群：** 工作器从共享队列拉取。

蜂群自动平衡负载；固定分配在分配任务慢时让快工作器空闲。

运行：

```
python3 code/main.py
```

输出显示每工作器任务数（蜂群分布不均但最优）和挂钟时间。

## 使用

`outputs/skill-swarm-fit.md` 评估任务是否应该使用蜂群 vs 主管。输入：任务独立性、时长方差、排序要求、可调试性需求。

## 交付

清单：

- **带老化的优先级队列。** 防止长任务饥饿。
- **工作器幂等性。** 任务可能被处理多次如果工作器中途崩溃。工作器必须幂等。
- **持久队列。** 生产使用 Kafka、Redis Streams 或数据库支持队列。`queue.Queue` 仅内存。
- **每个任务可观测性。** 每个任务有跟踪 ID；每个工作器记录开始/结束及它。
- **背压。** 如果队列增长快于工作器消耗，放慢生产者。

## 练习

1. 运行 `code/main.py`。在可变时长工作负载上蜂群比顺序快多少？比固定分配快多少？
2. 添加优先级队列变体（使用 `queue.PriorityQueue`）。按任务"重要性"字段分配优先级。在持续负载下观察低优先级任务是否饥饿。
3. 实现热点检测器：当任何工作器处理最慢工作器 3 倍以上任务时记录。这表明任务时长分布的什么？
4. 阅读 Matrix 论文摘要和第 3 节。识别 Matrix 接受（可扩展性增益）的一个具体权衡和放弃（可追溯性、确定性）的一个。
5. 将蜂群演示转换为使用 (task_type, payload) 元组队列，工作器只订阅特定类型。当任务异构时什么路由规则有意义？

## 关键术语

| 术语 | 人们常说 | 实际含义 |
|------|----------------|------------------------|
| 蜂群架构 | "去中心化智能体" | 工作器从共享队列拉取；无中央编排器。 |
| 事件总线 | "智能体订阅主题" | 消息代理按类型或内容路由任务到工作器。 |
| 饥饿 | "任务永不运行" | 低优先级任务因高优先级工作不断到达而永不获取。 |
| 热点 | "一个工作器淹没" | 负载不平衡，一个工作器获取大部分任务。 |
| 背压 | "减慢生产者" | 队列满时发信号上游停止生产。 |
| 幂等工作器 | "安全重运行" | 任务处理两次产生相同结果。需要因为工作器可能中途崩溃。 |
| 持久队列 | "崩溃存活" | 磁盘或复制存储支持的队列；工作器崩溃时任务不丢失。 |
| Matrix 框架 | "全消息传递蜂群" | 数据和控制流都是分布式队列上的序列化消息。 |
