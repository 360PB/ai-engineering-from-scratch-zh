# 共享内存和黑板模式

> 2026 年多智能体系统中两种方法并存：**消息池**（每个人都看到每个人的消息，如 AutoGen GroupChat 或 MetaGPT）和**带订阅的黑板**（智能体订阅相关事件，如 Context-Aware MCP 或 Matrix 框架）。两者都是多智能体系统中唯一的有状态部分——这意味着两者都是有趣 bug 所在之地。参考失败模式是**记忆中毒**：一个智能体产生幻觉"事实"，其他智能体将其视为已验证，随着准确性在更长时间内逐渐衰减，比即时崩溃更难调试。本课从标准库构建两种结构，注入一次中毒攻击，并展示在生产中真正有效的三种缓解措施。

**类型：** 学习 + 构建
**语言：** Python（标准库，`threading`）
**前置知识：** Phase 16 · 04（原始模型），Phase 16 · 09（并行蜂群网络）
**时间：** 约 75 分钟

## 问题

多智能体系统需要一个地方让智能体共享事实。字面选项是"在消息中传递一切"——但这用额外复制重新发明了共享状态。另一个选项是"给每个人一个全局日志"——但全局日志无限增长且容易中毒。第三个是"为每个智能体投影一个视图"——可扩展但模式重。

当其中一个智能体产生幻觉并将幻觉写入共享状态时，每个读取该状态的下游智能体都将幻觉作为事实采纳。到人类注意到时，推理链已经深入五步，根因是第三条消息。从来没有智能体崩溃。没有测试失败。系统"工作了。"幻觉通过共享状态从一个智能体的上下文交叉到每个下游智能体的推理中。

这是记忆中毒。它是 MAST 分类法（Cemri 等，arXiv:2503.13657）中记录第二多的失败族，它是结构性的：任何没有出处和不可写验证器的共享内存设计最终都会表现出它。

## 概念

### 两种主要拓扑

**完整消息池。** 每个智能体读取每条消息。AutoGen GroupChat 和 MetaGPT 使用这种。简单、透明、可检查，但无法扩展到约 10 个智能体以上，因为每个智能体的上下文被其他智能体的工作填满。

```
agent-A ──write──▶ ┌────────────────┐ ◀──read── agent-D
                   │ message pool   │
agent-B ──write──▶ │                │ ◀──read── agent-E
                   │ (global log)   │
agent-C ──write──▶ └────────────────┘ ◀──read── agent-F
```

**带订阅的黑板。** 智能体声明对主题的兴趣；底层只路由相关消息。CA-MCP（arXiv:2601.11595）和 Matrix 去中心化框架（arXiv:2511.21686）使用这种。可扩展更远，但需要前期模式设计使订阅有意义。

```
                   ┌─ topic: prices ──┐
agent-A ──pub────▶ │                  │ ──▶ agent-D (subscribed)
                   ├─ topic: orders ──┤
agent-B ──pub────▶ │                  │ ──▶ agent-E (subscribed)
                   ├─ topic: alerts ──┤
agent-C ──pub────▶ │                  │ ──▶ agent-F (subscribed)
                   └──────────────────┘
```

### 各适合什么

- **完整池**在智能体少（< 10）、异构、对话短时域时胜出。每个人看到一切时推理谁说了什么很简单。
- **黑板**在智能体多、角色同构但实例众多（蜂群）、对话长期运行时胜出。路由节省 token 成本和上下文污染。

生产系统经常混合：在顶部使用小型完整池（规划层），在下方使用黑板（工作器层）。

### 记忆中毒，一个场景

三个智能体处理研究任务。智能体 A 是检索智能体。智能体 B 是总结器。智能体 C 是分析师。

1. A 获取一个页面并将消息写入共享状态："该研究报道了 42% 的准确率提升。"
2. 获取的页面实际说的是"4.2% 的提升。"A 产生了一个小数点的幻觉。
3. B，读取共享状态，写道："报道了巨大的 42% 准确率增益（来源：A）。"
4. C，读取共享状态，写道："建议采用——42% 的提升是变革性的。"
5. 最终报告引用了一个从未存在的 42% 数字。

没有智能体崩溃。没有测试失败。系统"工作了。"幻觉通过共享状态从一个智能体的上下文交叉到每个下游智能体的推理中。

### 为什么这是结构性的

没有共享状态，智能体 A 的幻觉留在 A 的上下文中。下游智能体将重新获取或重新派生，可能捕获错误。对于天真的共享状态，A 的上下文成为每个人的上下文，幻觉被洗成事实。

问题不在于共享状态本身——而是没有出处和独立验证器的共享状态。三种缓解措施解决此问题：

1. **在每次写入时记录出处。** 共享状态中的每个条目记录谁写了它、何时、在什么提示下、以及（如果有的话）智能体引用了什么源。下游智能体以与出处相关的怀疑度读取。
2. **版本化写入；将其视为仅追加。** 更正是引用被取代旧条目的新条目，而不是就地更新。审计跟踪被保留。
3. **保留至少一个不能写入共享状态的智能体。** 只读验证器智能体对条目进行采样，重新获取源，并标记不一致。因为它不能写入池，所以不能被池中毒。

### 黑板先例（Hayes-Roth，1985）

黑板模式比 LLM 智能体早四十年。Hayes-Roth（1985，"控制的黑板架构"）描述了观察全局黑板、贡献部分解决方案并触发其他源的专家知识源。2026 年的黑板（CA-MCP、Matrix）是同一模式，只是 LLM 智能体作为知识源，JSON blobs 作为部分解决方案。旧文献记录了写争用、机会控制和一致性的解决方案，现代系统重新发现。

### 投影 vs 完整视图

纯黑板给每个订阅者相同的投影（主题范围）。更激进的设计是**每个智能体投影**：每个智能体获得根据其角色定制的视图。LangGraph 的状态 reducer 是 2026 年的规范实现——reducer 函数将全局状态折叠成角色特定的切片。

每个智能体投影可扩展更远但需要一个模式。没有它，你在每个智能体的提示中重建临时投影。

### 写争用模式

多个智能体同时写入是一个并发问题，而不仅仅是 LLM 问题。三种模式有效：

- **顺序写入器（单生产者）。** 所有写入通过一个序列化协调器智能体。简单，但成为瓶颈。
- **带版本的乐观并发。** 每个条目有一个版本；写入者在版本不匹配时失败并重试。经典数据库技术。
- **主题分区。** 不同智能体拥有不同主题。没有跨主题争用。需要设计的分区边界。

大多数 2026 年框架默认为顺序写入器，因为 LLM 调用足够慢，争用很少发生，瓶颈不会造成伤害。

### 不可写验证器

最有承载力的缓解措施是只读验证器。实现规则：

- 验证器与团队共享状态（读取黑板或池）。
- 验证器没有共享状态的写句柄——只有单独的验证通道。
- 验证器独立获取写入中引用的源。标记不一致。
- 验证器自己的输出路由到人类或单独的决策智能体，从不反馈到池中。

没有这种分离，验证器的输出成为池中的新条目，这意味着被污染的池污染验证器，验证器污染其验证。

## 构建

`code/main.py` 用标准库 Python 实现两种拓扑，加上一个玩具中毒攻击和三种缓解措施。

- `MessagePool` — 线程安全的仅追加日志，带完整读取。
- `Blackboard` — 主题键控的发布/订阅，带每个智能体订阅。
- `ProvenanceEntry` — 每次写入记录（写入者、时间戳、提示哈希、源 URI）。
- `PoisoningScenario` — 运行一个三智能体研究任务，其中智能体 A 产生小数幻觉。打印最终报告。
- `Verifier` — 一个只读智能体，重新获取源并标记不一致。运行相同场景并带有验证器。

运行：

```
python3 code/main.py
```

预期输出：
- 运行 1（无验证器）：幻觉的 42% 传播到最终报告。
- 运行 2（有验证器）：验证器标记不一致，池被标记为"已标记"，最终报告包含撤回。

## 使用

`outputs/skill-memory-auditor.md` 是一个技能，审查任何多智能体系统的共享内存设计是否有出处、版本化和验证器分离。在生产之前在新多智能体架构上运行它。

## 交付

对于任何共享内存设计：

- 在每次写入时记录出处：`(writer, timestamp, prompt_hash, tool_calls_cited, source_uri)`。
- 使日志仅追加。更正是引用被取代条目的新条目。
- 部署至少一个带独立源访问的只读验证器智能体。
- 将验证器输出路由到单独通道，而不是反馈到共享池中。
- 记录被取代写入的比例——上升的比例是幻觉模式的早期证据。

## 练习

1. 运行 `code/main.py`。确认运行 1 传播幻觉，运行 2 捕获它。
2. 添加第二个幻觉：智能体 B 捏造一个数据集大小。验证器应该捕获两者，而无需为任何一个进行手动调优。
3. 将完整池切换到带主题分区（`prices`、`summaries`、`analyses`）的黑板。主题分区使哪些中毒场景更难实施，哪些没有帮助？
4. 阅读 Hayes-Roth（1985，"控制的黑板架构"）。识别论文中两个本课未讨论的 2026 系统将受益的控制模式。
5. 阅读 CA-MCP（arXiv:2601.11595）。将其共享上下文存储映射到 `code/main.py` 中的 MessagePool 或 Blackboard 类。CA-MCP 在其上添加了哪些原语？

## 关键术语

| 术语 | 人们常说 | 实际含义 |
|------|----------------|------------------------|
| 消息池 | "共享聊天历史" | 每个智能体读取的仅追加日志。完全透明，扩展性差。 |
| 黑板 | "共享工作空间" | 主题键控的发布/订阅。智能体订阅相关主题。可扩展更远。 |
| 出处 | "谁写了什么" | 每次写入的元数据：写入者、时间戳、提示、源。 |
| 记忆中毒 | "幻觉传播" | 一个智能体的错误进入共享状态，下游智能体将其作为事实采纳。 |
| 仅追加 | "无就地更新" | 更正是引用被取代的新条目。保留审计跟踪。 |
| 不可写验证器 | "独立审计员" | 只读智能体，重新获取源并标记不一致。 |
| 投影 | "范围视图" | 从全局状态计算的每个智能体视图。LangGraph reducer 是规范案例。 |
| 知识源 | "专家智能体" | Hayes-Roth 1985 年对黑板参与者的术语。 |
