# 生产扩展 — 队列、检查点、持久化

> 扩展到数千并发运行的多 Agent 系统需要**持久化执行**。LangGraph 运行时在每个超级步骤后写入检查点，以 `thread_id` 为 key（默认用 Postgres）；Worker 崩溃后释放租约，另一个 Worker 恢复执行。Agent 可以无限期休眠等待人工输入。**MegaAgent**（arXiv:2408.09955）每个 Agent 运行一个生产者-消费者队列，有三个状态（空闲 / 处理中 / 响应）和双层协调（组内聊天 + 组间管理员聊天）。**Fiber/async** 优于每个任务一个线程：线程在等待 token 时 99% 的时间闲置， fiber 在 I/O 上协作让出。反驳观点：Ashpreet Bedi 的"Scaling Agentic Software"主张**FastAPI + Postgres + 别无他物**直到负载证明需要更多——简单架构比预期走得更远。本节构建一个持久化检查点日志、每个 Agent 的工作队列（含状态转换）、async 对比 thread 的演示，并落实"从简单开始"的务实原则。

**类型：** 学习 + 动手实现
**语言：** Python（标准库、`asyncio`、`sqlite3`）
**前置知识：** 第 16 章 · 09（并行蜂群网络）、第 16 章 · 13（共享内存）
**时长：** 约 75 分钟

## 问题背景

一个多 Agent 原型系统在一台笔记本上用三个 Agent 的内存事件循环运行良好。转入生产：

- Agent 有时运行数小时（长时间研究、人工介入等待）。
- Worker 进程崩溃。重启后丢失状态。
- 峰值负载是平均的 10 倍；需要水平扩展。
- 用户按 Agent 运行付费；你需要精确一次语义来计费。

内存事件循环无法满足以上任何需求。你需要一个持久化执行层。2026 年 4 月的主流选项：

1. 带检查点的工作流引擎（Temporal、LangGraph 运行时）。
2. 消息队列 + 状态存储（Postgres + SQS/RabbitMQ）。
3. Actor 模型框架（MegaAgent 的每个 Agent 生产者-消费者模式）。
4. 手撸 FastAPI + Postgres（Bedi 的论点）。

本节构建每种模式的精简版。

## 概念

### 持久化执行，模式解析

持久化执行引擎在每个"步骤"（LangGraph 术语中的超级步骤）之后持久化完整的程序状态。崩溃时：

```
Worker 在步骤中间崩溃
  -> 租约超时
  -> 另一个 Worker 接管 thread_id
  -> 从最后一个检查点恢复
  -> 不产生重复副作用
```

这能工作的前提：
- **可序列化状态。** 所有 Agent 状态必须可持久化。带有活动数据库连接的函数闭包无法存活。
- **确定性恢复。** 给定相同状态和相同输入，Agent 产生相同动作（或委托给外部确定性 Oracle 处理 LLM 调用）。
- **幂等副作用。** 外部调用（工具调用、支付）必须幂等或使用去重 key。

LangGraph 在每个超级步骤后写检查点；Temporal 在每个活动后写；Restate 使用事件溯源日志。三者实现相同模式。

### LangGraph 运行时

每个 Agent 有 `thread_id`；状态是一个类型化字典；每个超级步骤向检查点表追加一行。恢复时，运行时从最后一个检查点重放，不是从头开始。Agent 可以 `interrupt()` 等待人工输入；运行时持久化并释放 Worker。输入到达后，任何 Worker 都可以恢复。

这是 2026 年 4 月的生产参考设计。

### MegaAgent 每个 Agent 的队列

arXiv:2408.09955 描述了一个规模实验：数千个并发 Agent 在一个集群中。架构：

```
Agent i:
  state ∈ {空闲, 处理中, 响应}
  in_queue   <- 发给 Agent i 的消息
  out_queue  -> 回复 + 副作用

协调器:
  组内聊天  （同一组的 Agent）
  组间管理员聊天  （高层路由）
```

双层协调让组内对话密集进行，组间保持稀疏——这是将成本控制在数千 Agent 线性增长的关键模式。

### Async vs 每个任务一个线程

LLM 调用是 I/O 密集型的。一个等待下一个 token 的线程 99% 的时间都在闲置。线程每个约消耗 1MB RAM；10000 个并发调用仅栈就要 10GB。

Fiber（Python `asyncio`、Go goroutine、Rust `tokio`）在 I/O 上协作让出。10000 个并发调用可以在一个进程内轻松容纳。在 LLM-Agent 规模上，async 不是优化——是架构。

例外：CPU 密集的后处理（嵌入计算、分词器优化）仍然需要线程或进程。将 I/O 层与 CPU 层分开。

### Bedi 的反驳

"Scaling Agentic Software"（Ashpreet Bedi，2026）认为大多数团队在测量负载之前就过度设计了。务实的默认方案：

- FastAPI + Postgres。
- 每个 Agent 运行是一行记录；状态通过乐观并发原地更新。
- 后台任务通过 `pg_notify` 或简单的 Celery Worker。
- 重试策略写在应用代码里。

负载在 ~100 并发 Agent 运行以下、管理任务可处理时，这通常就够了。等测量到不行了再升级。

原则：当你遇到简单架构解决不了的真实问题时，才采用持久化执行框架。过早采用会浪费时间在并不值得的仪式感上。

### 精确一次语义

对于付费的 Agent 运行，你需要"有效精确一次"（至少一次传递 + 幂等消费者）。工程手段：

- **每运行一个去重 key。** 每次副作用调用都带上它。
- **发件箱模式。** 副作用先写入一张表，另一个独立进程执行。两个步骤都是幂等的。
- **补偿事务。** 副作用成功但跟踪写入失败时，安排一次补偿。

这些是数据库工程模式，不是 LLM 特有的。LLM 的额外成本只是调用慢；其他都是标准分布式系统。

### 彩虹部署

Anthropic 的多 Agent 研究系统使用"彩虹部署"：多个版本的 Agent 运行时并发运行，这样长时间运行的 Agent 不必在每次代码部署时被杀死。新版本在部分流量上灰度；旧版本等其 Agent 完成后才退出。

这对于长时间运行的有状态系统是标准做法；2026 年的适配是 Agent 可能存活数小时，所以部署周期必须与之兼容。

### 生产检查清单

- 持久化状态（检查点、快照，或发件箱 + 可重放日志）。
- 幂等副作用。
- LLM 调用的 Async I/O 层。
- 至少一次传递 + 去重。
- 有状态工作负载的彩虹/灰度部署。
- 可观测性：每个 Agent 的追踪、超级步骤审计、重试计数器。

## 动手实现

`code/main.py` 实现：

- `CheckpointStore` — SQLite 支持的检查点日志，以 thread-id 为 key。每次超级步骤追加一行。
- `run_with_checkpoint(agent, thread_id)` — 模拟崩溃中途运行；第二个 Worker 从最后一个检查点恢复。
- `AgentQueue` — 每个 Agent 的空闲 / 处理中 / 响应状态机，带小型工作队列。
- `demo_async_vs_threads()` — 通过 asyncio 和线程分别运行 500 个并发模拟"LLM 调用"；报告墙上时钟时间和峰值内存（近似值）。

运行：

```
python3 code/main.py
```

预期输出：检查点恢复模拟崩溃后成功；async 版本在 < 1 秒内处理 500 个并发调用；线程版本耗时数秒且每个并发单元的内存使用量高几个数量级。

## 产出

标准生产加固：

- **从简单开始（Bedi 原则）。** FastAPI + Postgres 直到你测量到不行。
- **优化前先埋点。** 每次运行延迟直方图、每步耗时、重试计数、失败分类。
- **副作用用发件箱模式。** 支付和外部 API 调用尤其如此。
- **彩虹部署。** 部署期间绝不杀死进行中的 Agent 运行。
- **当你遇到具体问题时才采用持久化执行引擎（Temporal / LangGraph / Restate）：** 数小时人工介入等待、跨区域协调、复杂的重试/补偿策略。
- **I/O 层用 Async。** CPU 密集后处理才用线程。

## 练习

1. 运行 `code/main.py`。确认检查点恢复工作；测量 async 与线程并发差异。
2. 实现**发件箱**表：每个工具调用先写入发件箱，再由一个独立的协程/任务执行。重复运行该工具调用验证幂等性。
3. 模拟**彩虹部署**：两个并发运行时版本；将新的 thread_id 对半分给每个；确认旧版本上的进行中线程不被中断。
4. 阅读 LangGraph 运行时文档（链接见下）。识别哪些运行时功能在手撸 FastAPI + Postgres 版本中最难复制。这是你采用的理由，还是可以推迟？
5. 阅读 MegaAgent（arXiv:2408.09955）第 3 节。双层协调（组内 + 组间管理员聊天）有明确描述。草拟你如何将其映射到两类队列族的消息队列上。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 持久化执行 | "持久化程序状态" | 引擎在每个超级步骤后写状态；崩溃恢复是确定性的。 |
| 超级步骤 | "事务边界" | 检查点之间的最小工作单元。LangGraph 术语。 |
| thread_id | "Agent 运行标识符" | 绑定检查点和恢复逻辑的 key。 |
| 幂等性 | "可以安全重试" | 重复副作用与一次尝试产生相同结果。 |
| 发件箱模式 | "解耦副作用" | 将意图写入表；独立执行器执行并标记完成。 |
| 至少一次传递 | "可能有重复" | 消息队列语义；去重 key 使消费者有效实现精确一次。 |
| 彩虹部署 | "版本重叠运行" | 在长时间运行工作负载期间多个运行时版本并发。 |
| Async fiber | "协作让出" | 用户态并发；相比线程对 I/O 密集型负载成本极低。 |
| 检查点 | "状态快照" | 超级步骤边界处的序列化状态；恢复的 key。 |
