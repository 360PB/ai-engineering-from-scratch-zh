# 角色专业化——规划器、批评者、执行器、验证器

> 2026 年最常见的多智能体分解：规划器、执行器、批评者。工作器做执行，规划器做计划，批评者做判断，验证器做测试。MetaGPT（arXiv:2308.00352）将此格式化为 SOP 编码到角色提示——产品经理、架构师、项目经理、工程师、质量工程师——遵循 `Code = SOP(Team)`。ChatDev（arXiv:2307.07924）通过"沟通去幻觉"链接设计者、程序员、审查员、测试员。验证器是承重结构：Cemri 等人（MAST，arXiv:2503.13657）显示每个多智能体失败都可追溯到缺失或破碎的验证。PwC 报告在 CrewAI 中结构化验证循环使准确率从 10% 提升到 70%——7 倍增益来自一个角色。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置知识：** Phase 16 · 04（原语模型），Phase 16 · 05（主管）
**时间：** 约60分钟

## 问题

通用多智能体系统产生通用输出。三个编码员在群聊中写出三种平淡代码。你添加更多智能体、更多轮次，仍不跨质量阈值。

修复不是更多智能体——是*不同*智能体。分配不同角色。给批评者规划器没有的工具。给验证者客观测试套件。现在系统有带根据纠正的内部分歧，不是并行猜测。

## 概念

### 四个规范角色

**规划器。** 读取目标，产生步骤列表或规范。工具：知识检索、文档。输出：结构化计划。

**执行器。** 一次读取一个计划步骤，产生产物。工具：实际工作工具（代码编译器、shell、API 客户端）。输出：产物。

**批评者。** 根据规划器意图读取执行器输出。工具：只读访问artifact、静态分析。输出：接受/拒绝，带原因。

**验证器。** 读取产物并运行确定性检查。工具：测试运行器、类型检查器、模式验证器。输出：通过/失败，带证据。

批评者是主观的、意见性的，通常基于 LLM。验证器是客观的、确定性的，通常基于代码。它们不是同一角色。

### MetaGPT 的 SOP 模式

MetaGPT（arXiv:2308.00352）将软件工程 SOP 编码为角色提示：

- **产品经理** 编写 PRD。
- **架构师** 产出系统设计。
- **项目经理** 拆分任务。
- **工程师** 实现。
- **QA 工程师** 运行测试。

每个角色有严格的输入/输出模式。角色提示说明角色*是什么*和角色*必须产生什么*。`Code = SOP(Team)` 公式——确定性 SOP 将一组 LLM 变成可预测流水线。

### ChatDev 的沟通去幻觉

ChatDev 添加了关键动作：当执行器需要计划中没有的具体细节时，在继续之前明确询问设计者。这防止了 plausible 发明细节的经典 LLM 失败。

实现：角色提示包括"当你需要未给你的具体信息时，在产生输出之前按名称询问相关角色。"

### 为什么验证者最重要

Cemri 等人（MAST）在 1642 条执行跟踪上追踪多智能体失败。21.3% 是验证缺口——系统交付了没人检查的答案。其余 79% 通常追溯到"有检查但静默失败或从未运行"。验证是承重角色。

PwC 报告（CrewAI 部署，2025 年）添加结构化验证循环将准确率从 10% 移到 70%。7 倍增益来自一个角色。

### 批评者 vs 验证者

- 批评者是 LLM 审查产物质量。主观的。会被 plausible 文本愚弄。
- 验证器是在产物上运行的确定性程序。客观的。给出带证据的通过/失败。不能被愚弄。

两者都用。批评者捕获验证器无法表述的品味问题。验证器捕获批评者看不到的 bug，因为它们只在运行时出现。

### 反模式

你系统中的每个角色都是 LLM，每个角色的输出都是"看起来不错"。经典 MAST 失败模式。添加至少一个其通过/失败由代码而非 LLM 决定的验证器。

### 框架映射

- **CrewAI** — `Agent(role, goal, backstory)` 是教科书专业化面。
- **LangGraph** — 节点可以有专业化提示；边强制流水线。
- **AutoGen** — 角色特定 ConversableAgent，组聊中用单字名称。
- **OpenAI Agents SDK** — 交接工具在角色专业化 Agent 之间。

## 构建

`code/main.py` 实现构建简单 Python 函数的 4 角色流水线：

- **规划器** 产生规范。
- **执行器** 生成代码字符串。
- **批评者**（LLM 模拟）标记明显问题。
- **验证器** 在沙盒中（`exec`）对测试用例运行生成代码。

演示运行两次：一次执行器产生正确代码（批评者 + 验证者都通过），一次执行器产生不规范代码（批评者漏掉 bug 因为看起来 plausible，验证器捕获因为测试失败）。

运行：

```
python3 code/main.py
```

## 使用

`outputs/skill-role-designer.md` 获取任务并产生角色名册（3-5 个角色）、每个角色的输入/输出模式，以及验证器检查。在将智能体接入框架之前使用。

## 交付

清单：

- **至少一个确定性验证器。** 永远不全 LLM。
- **每个角色显式 I/O 模式。** 规划器返回规范，不是 prose；执行器读取该模式。
- **沟通去幻觉。** 执行器必须在信息缺失时询问规划器；永不凭空发明。
- **批评者/验证器顺序。** 先运行批评者（便宜，捕获设计问题），后验证器（慢，捕获 bug）。
- **循环预算。** 最多 2 个批评者-执行者修改轮，然后升级给人类。

## 练习

1. 运行 `code/main.py` 并观察验证器如何捕获批评者漏掉的 bug。添加静态分析检查（计数 `return` 出现次数）作为额外验证器。它捕获运行时测试漏掉什么？
2. 添加第五个角色："需求分析师"将用户愿望翻译为规划器就绪规范。什么沟通去幻觉请求应该流向它？
3. 阅读 MetaGPT 第 3 节（"Agents"）。列出 MetaGPT 5 个角色中每个的输入/输出模式。
4. 阅读 ChatDev 的聊天链图（arXiv:2307.07924 图 3）。识别沟通去幻觉打破的循环，否则会无限。
5. PwC 的 7 倍准确率增益来自验证循环。假设三个任务添加验证器没有帮助——其中确定性正确性检查不可能或成本过高。

## 关键术语

| 术语 | 人们常说 | 实际含义 |
|------|----------------|------------------------|
| 角色专业化 | "不同智能体，不同工作" | 专为规划器/执行器/批评者/验证器角色调优的不同系统提示。 |
| SOP 模式 | "编码标准操作程序" | MetaGPT 框架：严格 I/O 模式 per 角色将团队变成流水线。 |
| 沟通去幻觉 | "先问后发明" | ChatDev 模式：执行器在缺失细节时询问规划者，不凭空捏造。 |
| 批评者 | "LLM 审查员" | 主观的、意见性的审查员。捕获品味问题。会被 plausible 文本愚弄。 |
| 验证器 | "确定性检查" | 基于代码的通过/失败。测试运行器、类型检查器、模式验证器。不能被愚弄。 |
| 验证缺口 | "没人检查" | MAST 失败的 21.3%。答案发出前没有会捕获 bug 的检查。 |
| 修改循环 | "批评者发回" | 批评者拒绝触发执行器带反馈重新运行。需要预算。 |
| 全 LLM 反模式 | "看起来不错" | 每个角色都是 LLM，没有确定性检查。经典 MAST 失败。 |
