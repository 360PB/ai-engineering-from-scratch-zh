# 主管 / 编排器-工作器模式

> 一个主导智能体规划并委托；专业工作器在并行上下文中执行并报告。这是 Anthropic 研究系统（Claude Opus 4 作为主导，Claude Sonnet 4 作为子智能体）背后的模式，在内部研究评估中测得比单 Opus 4 高 +90.2%。Anthropic 的工程帖子报告了 BrowseComp 方差的 80% 仅由 token 使用量解释——多智能体胜出主要是因为每个子智能体获得新鲜的上下文窗口。本课从原语构建主管模式，并涵盖 2026 年生产部署的工程经验。

**类型：** 学习 + 构建
**语言：** Python（标准库、`threading`）
**前置知识：** Phase 16 · 04（原语模型）
**时间：** 约75分钟

## 问题

研究是单智能体系统失败的原型任务。你问"2023 到 2026 年之间多智能体系统发生了什么变化？"单智能体依次读取五篇论文，将一半上下文填满它们的文本，然后不得不一起推理它们。它在读到第五篇时忘记了第一篇。它无法并行化。

主管模式修复了这个问题：一个主导智能体规划搜索，将每个子问题委托给工作器，并综合。每个工作器在自己的 200k token 窗口中处理一个狭窄问题。主导从不看到原始论文——只有工作器摘要。

Anthropic 的生产研究系统在内部研究评估中报告了 +90.2% 比单 Opus 4。同帖子注意到 BrowseComp 方差的 80% 仅由*token 使用量*解释。新鲜上下文每个子智能体是主要机制。

## 概念

### 模式

```
                 ┌──────────────┐
                 │   主导       │  规划、分解、
                 │  (Opus 4)   │  综合
                 └──┬────┬───┬──┘
                    │    │   │
            ┌───────┘    │   └───────┐
            ▼            ▼           ▼
      ┌─────────┐  ┌─────────┐  ┌─────────┐
      │ 工作器 1 │  │ 工作器 2 │  │ 工作器 3 │
      │(Sonnet) │  │(Sonnet) │  │(Sonnet) │
      └─────────┘  └─────────┘  └─────────┘
         新鲜        新鲜        新鲜
         上下文       上下文       上下文
```

主导从不读取原材料。工作器在主导综合之前看不到彼此的工作。每个箭头是一个狭窄产物的交接。

### 为什么它胜出

三个机制：

1. **每个子智能体的新鲜上下文。** 探索"FIPA-ACL 遗产"的工作器不携带主导花费在规划的 40k tokens。它为一个问题获得 200k 窗口。
2. **通过提示专业化。** 主导的提示是"分解和综合"，不是"研究"。每个工作器的提示是狭窄的："查找 X 发生了什么变化。"专注提示产生专注输出。
3. **并行性。** 工作器并发运行。挂钟时间大致是 `max(工作器时间) + 规划 + 综合`，不是 `sum(工作器时间)`。

### 工程经验（Anthropic 2025）

Anthropic 帖子列出了仍然 2026 年相关的几条生产经验：

- **努力与查询复杂性相称。** 简单查询：一个智能体，3-10 次工具调用。复杂查询：10+ 个智能体。主导必须估计，不是调用方。
- **先宽后窄。** 先分解为宽子问题，然后如果答案需要深度，为每个子问题生成更多工作器。
- **彩虹部署。** 智能体是长运行和有状态的。传统蓝绿不工作。Anthropic 使用彩虹：新版本逐渐推出而旧版本排出。
- **Token 使用量占主导。** 多智能体是单智能体约 15 倍 tokens。只有任务价值正当时才运行它。

### LangGraph 的转变

LangGraph 最初发布了一个带有高级 `create_supervisor` 帮助函数的 `langgraph-supervisor` 库。2025 年 LangChain 将推荐改为直接通过工具调用实现主管模式，因为工具调用对*主管看到什么*有更多控制（上下文工程）。库仍然工作；文档现在推荐工具调用形式。

### 失败模式

- **主导幻觉计划。** 如果主导生成的子问题不能分解真正的问题，工作器在错误目标上做精确研究。
- **工作器过度探索。** 没有明确范围边界，工作器偏离其分配的子问题并污染综合步骤。
- **综合冲突。** 两个工作器返回矛盾事实。主导必须重新询问（加一轮）或明确注意分歧。悄悄选择一方是最坏失败：用户永远不知道发生了分歧。

### 何时主管模式错误

- **顺序任务。** 如果步骤 2 确实需要步骤 1 的输出，并行性没有好处。使用流水线（CrewAI 顺序、LangGraph 线性图）。
- **简单查询。** 单智能体处理更快更便宜。在生成工作器之前使用主导的"努力规模"检查。
- **严格确定性。** 主管使用 LLM 选择委托。静态图在审计/重放更重要时更好。

## 构建

`code/main.py` 使用 `threading` 实现三个并行工作器的主管。主导分解查询为子问题，工作器并发运行每个子问题，主导综合。没有真实 LLM——工作器是脚本化的，模拟获取和总结。

关键结构：

- `Lead.plan(query)` 将查询拆分为 3 个子问题。
- `Worker.run(sub_q)` 返回假摘要（生产中可以是任何工具使用智能体）。
- `Lead.run(query)` 启动线程中的工作器，join，并综合。

运行：

```
python3 code/main.py
```

输出显示计划、并行工作器跟踪（带开始/结束时间戳）和最终综合。你可以看到挂钟收益：三个 0.3 秒工作器在约 0.35 秒内运行，不是 0.9。

## 使用

`outputs/skill-supervisor-designer.md` 获取用户查询并生成主管模式设计：主导系统提示、工作器角色、子问题分解规则和综合模板。在构建新的研究风格智能体系统之前使用它。

## 交付

部署主管模式前的清单：

- **模型配对。** 主导用推理层模型（Opus 级，`o3` 级）。工作器用更快更便宜的模型（Sonnet，`o4-mini`）。
- **工作器超时。** 任何超过 2× 中位运行时间的工作器被终止；主导用更窄范围重新生成或不使用它继续。
- **每个工作器 token 上限。** 硬限制（比如说 10× 预期综合输入）防止失控工作器超出预算。
- **可观测性。** 跟踪主导的计划、每个工作器的工具调用和综合。这是任何事后调试的基础。
- **彩虹推出。** 有状态长运行智能体需要渐进版本转换，不是热交换。

## 练习

1. 运行 `code/main.py`，然后修改主导生成 5 个工作器而不是 3 个。观察挂钟效应。在什么工作器数量下生成开销超过并行节省？
2. 实现工作器超时：终止运行超过 0.5 秒的任何工作器，让主导综合剩余结果。你需要什么可观测性来知道工作器被切断了？
3. 添加冲突检测到主导综合：两个工作器返回矛盾答案时，主导注意分歧而不是选择一方。在不调用 LLM 的情况下你如何检测矛盾？
4. 阅读 Anthropic 的研究系统工程帖子。列出这个玩具演示在生产中运行需要采用的三条实践。
5. 比较 LangGraph 的 `create_supervisor`（遗留）与新的工具调用推荐。哪个给你更好的控制主管看到什么？为什么 Anthropic 明确只传递子答案而不传递原始工作器上下文到综合？

## 关键术语

| 术语 | 人们常说 | 实际含义 |
|------|----------------|------------------------|
| 主管 | "主导智能体" | 规划、委托和综合的编排智能体。不自己做工作。 |
| 工作器 | "子智能体" | 由主管调用的专注智能体，具有狭窄范围和自己的上下文窗口。 |
| 编排器-工作器 | "主管模式" | 同一事物的不同名称。2026 年文献两者都用。 |
| 新鲜上下文 | "干净窗口" | 工作器上下文从其系统提示和分配的问题开始，不是主导的历史。 |
| 彩虹部署 | "渐进推出" | 长运行有状态智能体需要版本化排出和替换，不是蓝绿。 |
| Token 主导 | "上下文是变量" | 80% 的研究评估方差来自总使用 tokens 量，不是模型选择 per Anthropic。 |
| 努力规模 | "匹配智能体数量到复杂性" | 主导估计查询难度，相应生成 1 vs 10+ 工作器。 |
| 综合冲突 | "工作器不同意" | 两个工作器返回矛盾事实；主导必须注意分歧，不是悄悄选择一方。 |
