# 层级架构及其失败模式

> 层级是嵌套主管。管理者智能体之上有子管理者之上有工作器。CrewAI `Process.hierarchical` 是教科书版本：一个 `manager_llm` 动态委托任务并验证输出。LangGraph 等价物是 `create_supervisor(create_supervisor(...))`。当任务真实对应组织架构时这是自然模式。也是最可能崩溃到管理者循环的模式——管理者智能体分配工作不佳、误解子输出或无法达成共识。顺序通常胜过它。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置知识：** Phase 16 · 05（主管模式）
**时间：** 约60分钟

## 问题

一旦主管模式理解了，自然的下一步是"如果工作器本身就是主管呢？"团队有子团队；公司有部门中的部门。层级架构镜像了那个。

问题：LLM 管理者不等于人类管理者。人类管理者对报告内容有稳定先验。LLM 管理者每轮从其上下文中的任何内容重新推理组织。那个上下文中的微小漂移，整个树就错误分配工作了。

## 概念

### 形状

```
                 管理者
                 ┌─────┐
                 └──┬──┘
           ┌────────┴────────┐
           ▼                 ▼
       子管理者 A         子管理者 B
       ┌─────┐           ┌─────┐
       └──┬──┘           └──┬──┘
         ┌┴──┬──┐          ┌┴──┐
         ▼   ▼  ▼          ▼   ▼
       W1  W2  W3         W4  W5
```

每个内部节点规划、委托和综合。只有叶子做工作。

### 适用场景

- **清晰组织映射。** 如果真实任务是部门性的（"法务审查文档，财务审查文档，工程审查文档，然后总结给高管"），层级是显式的。
- **本地综合。** 每个子管理者在其团队输出综合后才让顶级管理者看到。顶级管理者看到三个子管理者摘要，不是十五个工作器输出。

### 适用场景

三个 2026 年事后分析不断发现的失败模式：

1. **任务分配错误。** 管理者读取目标，幻觉分解，委托给错误的子管理者。因为子管理者忠实地按给定工作，错误只在顶部综合时表面——距离人类本可捕获的地方一层。
2. **输出误解。** 子管理者返回"无法验证声明 X"。顶级管理者综合为"声明 X 未确认"。每层含义漂移。
3. **共识循环。** 两个子管理者不同意；顶级管理者要求他们协调；他们向下重新委托；工作器重新运行；子管理者返回略微不同的答案；循环。CrewAI 的 `Process.hierarchical` 用步数限制防范这个，但限制本身现在是超参数。

### 决定问题

顺序（线性流水线）vs 层级：你的任务真实有独立子团队，还是一个线性流假装是树？如果是后者，使用顺序。如果是前者，使用层级但预算明确协调规则。

### CrewAI 的实现

`Process.hierarchical` 连接一个管理者 LLM 在专家团队之上。管理者：

- 接收顶级任务，
- 分配子任务给团队，
- 评估团队输出，
- 决定是接受、重新委托还是迭代。

文档：https://docs.crewai.com/en/introduction（在核心概念下找"Hierarchical Process"）。

### LangGraph 的实现

LangGraph 使用嵌套 `create_supervisor` 调用。内部主管有自己的图；外部主管将内部图视为不透明节点。这比 CrewAI 更干净用于调试（你可以逐步查看每个图），但用动态重塑树来表达更困难。

参考：https://reference.langchain.com/python/langgraph-supervisor。

## 构建

`code/main.py` 运行 3 级层级：

- 顶级管理者：将任务分为"工程"和"法律"分支，
- 工程子管理者：分为"前端"和"后端"工作器，
- 法律子管理者：一个工作器。

演示对比快乐路径（每个人都同意）和**扰动路径**顶级管理者的分解将"法律"误标记为"财务"，观察错误级联——子管理者忠实地做财务工作，顶级综合器报告财务发现，原始法律问题未回答。

运行：

```
python3 code/main.py
```

输出显示两条路径，清晰并排"被要求什么"vs"交付了什么"。

## 使用

`outputs/skill-hierarchy-fitness.md` 评估给定任务是否应该使用层级、顺序或扁平主管。输入：任务描述、组织结构、协调预算。输出：模式推荐及要防范的具体失败模式。

## 交付

如果你交付层级：

- **树深度上限为 2。** 三级已经隐藏大多数错误从可观测性。
- **明确协调预算。** 在顶级管理者必须提交之前的最大轮数。通常 2。
- **每个综合的来源。** 每个节点的摘要必须引用产生它的叶子输出。
- **分解漂移警报。** 记录管理者每步的分解；与用户查询 diff。如果分解不再覆盖查询，触发警报。

## 练习

1. 运行 `code/main.py` 并比较快乐 vs 扰动。在顶部输出完全偏离用户问题之前需要多少级管理者交接？
2. 添加第三级（top → 子 → 子子 → 工作器）。测量扰动路径纠正自己 vs 随着深度增长完全偏离的频率。
3. 在每个子管理者实现一个"金丝雀"工作器，总是被问原始用户问题不变。使用金丝雀答案检测分解漂移。当金丝雀与综合答案不同意时管理者应该如何反应？
4. 阅读 CrewAI 的 `Process.hierarchical` 文档。识别 CrewAI 应用的一条具体 guardrail（步数限制、manager_llm 约束）并描述它针对哪个失败模式。
5. 比较嵌套 LangGraph 主管与 CrewAI 层级。哪个使协调循环更便宜检测？

## 关键术语

| 术语 | 人们常说 | 实际含义 |
|------|----------------|------------------------|
| 层级 | "组织图模式" | 主管之上有主管；只有叶子做工作。 |
| 管理者 LLM | "老板" | 在内部节点分解、分配和验证的 LLM。 |
| 分解漂移 | "老板丢失了主题" | 顶级管理者的分解不再覆盖原始问题。 |
| 协调循环 | "无尽会议" | 子管理者不同意；顶级重新委托；工作器重新运行；循环直到预算耗尽。 |
| 深度 2 天花板 | "不要深于 2 级" | 经验法则：3+ 级崩溃可观测性。 |
| 金丝雀问题 | "每级的事实真相" | 总被问原始查询不变的工作器，检测漂移。 |
| 来源链 | "谁说了什么" | 从每个综合追溯到产生它的叶子输出的跟踪。 |
