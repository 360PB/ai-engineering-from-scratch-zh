# 混合记忆：向量 + 图 + KV（Mem0）

> Mem0（Chhikara 等，2025）将记忆视为三个并行存储——向量用于语义相似性，KV 用于快速事实查询，图用于实体关系推理。检索时由一个评分层融合三者。这是 2026 年生产级外部记忆的标准架构。

**类型：** 动手实现
**语言：** Python（标准库）
**前置知识：** 第 14 阶段 · 07（MemGPT），第 14 阶段 · 08（Letta Blocks）
**学习时长：** 约 75 分钟

## 学习目标

- 解释为什么单一存储（仅向量、仅图、仅 KV）不足以支撑 Agent 记忆。
- 说出 Mem0 的三个并行存储及其各自的优化目标。
- 描述 Mem0 的融合评分——相关性、重要性和时效性——以及为什么它是加权求和而非层级结构。
- 用标准库实现一个三存储记忆玩具，包含写入三个存储的 `add()` 和融合结果的 `search()`。

## 问题背景

单一存储在三类查询中必有一类表现不佳：

- **语义相似性** —— "上周我们讨论了什么关于 Agent 漂移的事？"向量胜出；KV 和图都漏掉。
- **事实查询** —— "用户的电话号码是什么？"KV 胜出；向量浪费，图杀鸡用牛刀。
- **关系推理** —— "哪些客户共享同一计费主体？"图胜出；向量和 KV 无法回答。

生产环境中的 Agent 在同一会话内会发出全部三类查询。单一存储永远有两类表现不佳。Mem0 的贡献是将三者统一到一个 `add`/`search` 接口下，用评分函数完成融合。

## 核心概念

### 三个存储并行写入

Mem0（arXiv:2504.19413，2025 年 4 月）在 `add(text, user_id, metadata)` 时：

1. 从文本中提取候选事实（由 LLM 驱动）。
2. 将每个事实写入向量存储（嵌入向量）用于语义搜索。
3. 将每个事实以 `(user_id, fact_type, entity)` 为键写入 KV 存储，实现 O(1) 查询。
4. 将每个事实以带类型边的形式写入图存储（Mem0g），用于关系查询。

在 `search(query, user_id)` 时：

1. 向量存储返回按嵌入余弦相似度排序的 top-k 结果。
2. KV 存储根据查询派生出的 `(user_id, type, entity)` 返回精确命中。
3. 图存储返回从查询实体可达的子图。
4. 评分层融合三者的结果。

### 融合评分

```
score = w_relevance * relevance(q, record)
      + w_importance * importance(record)
      + w_recency * recency(record)
```

- **相关性（Relevance）** — 向量余弦相似度、KV 精确匹配、图的路径权重。
- **重要性（Importance）** — 写入时打标签或学习得到（某些事实权重更高：姓名、ID、策略）。
- **时效性（Recency）** — 自上次写入或读取以来的时间按指数衰减。

权重按产品调优。聊天类 Agent 适合更高的 `w_recency`；合规类 Agent 适合更高的 `w_importance`；检索类 Agent 适合更高的 `w_relevance`。

### Mem0g 与时序推理

Mem0g 增加了冲突检测器。当新事实与已有边矛盾时，不删除旧边而将其标记为无效。时序查询（"用户 3 月份在哪个城市？"）在有效时间内子图上遍历。

这是 Letta 失效模式在合规级场景下的泛化。

### 基准数据

Mem0 论文报告（2025 年）：

- **LoCoMo**（长对话记忆）：91.6
- **LongMemEval**（长时序情景记忆）：93.4
- **BEAM 1M**（100 万 token 记忆基准）：64.1

对比基线（全上下文 128k LLM、扁平向量存储、扁平 KV）均落后 10 分以上。基准分本身不能决定选型——看运维形态才能——但数据说明融合设计并非可有可无。

### 作用域分类

Mem0 按作用域拆分记忆：

- **用户记忆** — 跨会话持久化，以 `user_id` 为键。
- **会话记忆** — 仅在单个线程内持久化。
- **Agent 记忆** — 每个 Agent 实例的状态。

每次写入选择一个作用域。检索时可跨作用域查询并对各作用域加权。不经思考混用作用域会导致 "助手把 Bob 的项目告诉了 Alice" 的事故。

### 该模式的常见失效

- **嵌入漂移。** 前一百条查询看起来正确，但随着语料库增长，向量结果会退化。需要定期对使用频率最高的 N 条记录重新嵌入。
- **KV 类型失控。** `(user_id, type, entity)` 看起来简单，但每个团队都会加上自己的 `type`。每季度审查一次类型集合。
- **图爆炸。** 一个噪声提取器每条消息产生 50 条边。对每次 `add` 调用的图写入设置上限；丢弃置信度低的边。

## 动手实现

`code/main.py` 用标准库实现三存储模式：

- `VectorStore` — 朴素 token 重叠相似度（作为嵌入的替代）。
- `KVStore` — 以 `(user_id, fact_type, entity)` 为键的字典。
- `GraphStore` — 带类型的边（subject, relation, object, valid）。
- `Mem0` — 顶层门面，提供 `add()`、`search()`、融合评分和作用域感知检索。
- 一个多用户、多会话对话的完整运行示例。

运行：

```
python3 code/main.py
```

输出展示三条独立召回路径以及融合后的 top-k。在 `main()` 顶部调整评分权重，观察排名变化。

## 用现成库

- **Mem0（Apache 2.0）** — 生产可用。自托管可用 Postgres + Qdrant + Neo4j，或使用托管云服务。
- **Letta** — 三层核心/召回/归档；可对接你自己的向量和图后端。
- **Zep** — 商业替代方案，带时序知识图谱和事实提取。
- **自建方案** — 当需要对提取器（合规场景）或融合权重（时效性主导的语音 Agent）有精确控制时。

`outputs/skill-hybrid-memory.md` 生成一个三存储记忆骨架，含融合评分器、作用域分类和时序失效机制。

## 练习

1. 用真实嵌入模型（sentence-transformers、Ollama、OpenAI embeddings）替换玩具向量相似度。在合成长对话上测量 recall@10。随着写入超过 1000 条，排名是否漂移？
2. 增加时序查询：`search(query, as_of=timestamp)`，仅返回该时间点及之前有效的记录。哪个存储改动最大？
3. 实现冲突检测器：如果新事实与图中的边矛盾，使旧边失效并记录两条边。在"用户住在柏林" → "用户住在里斯本"上测试。
4. 将用户反馈维度（对检索结果的 thumbs-up）加入融合评分器。如何防止刷分（Agent 只返回它本来就喜欢的记录）？
5. 阅读 Mem0 文档（`docs.mem0.ai`）。将玩具实现改为 `mem0` 客户端调用。在同一组 20 个测试查询上比较检索质量。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| Hybrid memory（混合记忆） | "向量+图+KV" | 三个存储并行写入，检索时融合 |
| Fact extraction（事实提取） | "记忆摄取" | LLM 将文本拆解为（实体，关系，事实）元组 |
| Fusion scoring（融合评分） | "相关性排序" | 相关性、重要性、时效性的加权求和 |
| Scope（作用域） | "记忆命名空间" | user / session / agent — 决定谁能看到什么 |
| Mem0g | "记忆图" | 带时序有效性的类型边，用于关系查询 |
| Temporal invalidation（时序失效） | "软删除" | 将矛盾的边标记为无效；永不物理删除 |
| Embedding drift（嵌入漂移） | "检索退化" | 向量质量随语料库增长而下降；需定期重新嵌入 |
