# 基于 HTN 与演化搜索的规划

> 符号规划处理的是"方案在构造上可证明正确"的情形。演化代码搜索处理的是"Fitness 函数可机器校验"的情形。ChatHTN（2025）和 AlphaEvolve（2025）展示了各自在与 LLM 配对时的能力边界。

**类型：** 动手实现
**语言：** Python（标准库）
**前置要求：** 第 14 阶段 · 02（ReWOO 与 Plan-and-Execute）
**时长：** 约 75 分钟

## 学习目标

- 解释层级任务网络：任务、方法、算子、前置条件、效果。
- 描述 ChatHTN 的混合循环——符号搜索配合 LLM 后备分解。
- 解释 AlphaEvolve 的演化循环，以及为何它只能在有程序化评估器的条件下工作。
- 用标准库实现一个 HTN 规划器的玩具版本和一个演化搜索的玩具版本。

## 问题所在

ReWOO（第 02 课）、Plan-and-Execute 与 ReAct 覆盖了大多数 Agent 规划场景，但有两个例外它们处理不好：

1. **需要可证明正确性的方案。** 调度、航线规划、合规工作流——方案必须在构造上是合理的。让 LLM 规划器靠流式输出来生成方案，偶尔会在某一步产生幻觉，这是不可接受的。
2. **目标函数可机器校验的优化问题。** 矩阵乘法、调度启发式、编译器 pass——目标不是"一个正确的方案"，而是"最好的方案"。

HTN 规划与 AlphaEvolve 分别解决这两类不同的问题。两者都将 LLM 用作放大器，而非替代品。

## 核心概念

### 层级任务网络

HTN 由以下要素组成：

- **任务**——复合任务（待分解）和原子任务（可直接执行）。
- **方法**——将复合任务分解为子任务的方式，带前置条件。
- **算子**——原子动作，带前置条件和效果。
- **状态**——一组事实。

规划过程：给定一个目标任务和初始状态，找到一种分解方式，将目标任务分解为原子算子序列，且序列中每个算子的前置条件在当前状态下均满足。

HTN 出现远早于 LLM，至今仍是可证明正确方案领域的参考标准。

### ChatHTN（Gopalakrishnan 等，2025）

ChatHTN（arXiv:2505.11814）将符号 HTN 与 LLM 查询交织在一起：

1. 尝试用已有方法分解当前复合任务。
2. 如果没有方法适用，问 LLM："在状态 `s` 下，如何分解任务 `task`？"
3. 将 LLM 的回答翻译为候选子任务。
4. 对照算子模式验证，拒绝无效的分解。
5. 递归。

论文的核心主张：生成的每个方案都是可证明合理的，因为 LLM 的建议仅作为候选分解方案进入，始终不作为直接方案修改。符号层负责正确性；LLM 扩展方法库。

在线方法学习（OpenReview `gwYEDY9j2x`，2025 后续工作）增加了一个学习器，通过回归泛化 LLM 生成的分解方案——将 LLM 查询频率降低最多 75%。

### AlphaEvolve（Novikov 等，2025）

AlphaEvolve（arXiv:2506.13131，DeepMind，2025 年 6 月）是另一种动物：由 Gemini 2.0 Flash/Pro 组合编排的演化代码搜索。

循环如下：

1. 以一个种子程序加一个程序化评估器启动（评估器返回一个 fitness 分数）。
2. LLM 组合提出变异。
3. 让变异通过评估器。
4. 保留最优；继续变异。

已发表的成果：

- 56 年来首次改进 Strassen 的 4×4 复数矩阵乘法（48 次标量乘法）。
- 通过 Borg 调度启发式恢复了 0.7% 的 Google 计算资源。
- 在前沿工作负载上实现了 32% 的 FlashAttention 加速。

关键约束：Fitness 函数必须可机器校验。对文本答案做演化搜索不会收敛。

### 何时用哪个

| 问题类型 | 推荐方案 | 原因 |
|------------|---------|------|
| 有硬约束的调度 | HTN + ChatHTN | 可证明合理性 |
| 编译器优化 | AlphaEvolve | 可机器校验的 Fitness |
| 多步任务执行 | ReAct / ReWOO | LLM 在循环中，无需形式保证 |
| 有测试的代码改进 | AlphaEvolve | 测试即评估器 |
| 策略约束型自动化 | HTN | 前置条件编码了策略 |

### 这种模式的失败场景

- **没有算子的 HTN。** 没有前置条件/效果模式，正确性主张就崩溃。ChatHTN 的"LLM 建议分解"依赖模式来拒绝无效操作。
- **没有真实评估器的 AlphaEvolve。** "让 LLM 判断代码是否更好"不是 Fitness 函数。评估器必须是确定性的且快速。
- **过度工程。** 大多数 Agent 任务两者都不需要。先用 ReAct 或 ReWOO。

## 动手实现

`code/main.py` 实现了两个玩具：

- 一个标准库 HTN 规划器，带算子、方法、前置条件、效果，以及一个 `LLMFallback`——当复合任务没有方法匹配时触发。"LLM" 是一个脚本化的分解器，使规划器可以离线运行。
- 一个标准库演化搜索，在算术程序空间中搜索：生成表达式，使输出在测试集上最小化 `|f(x) - target|`。评估器是确定性的。

运行：

```
python3 code/main.py
```

跟踪信息显示 HTN 规划器分解一个复合任务（中途有一次 LLM 后备），以及演化循环收敛到一个目标表达式。

## 用现成库

- **HTN 规划器**——`pyhop`、`SHOP3`，或为自己的领域定制一个，用于特定领域的策略执行。
- **ChatHTN**——研究代码；该模式（符号 + LLM 后备）可干净地移植到任何 HTN 规划器。
- **AlphaEvolve**——DeepMind 论文；该模式（组合 + 评估器）可复现。OpenEvolve 等开源分支正在出现。
- **Agent 框架**——目前尚无框架自带一流的 HTN 或 AlphaEvolve。作为子 Agent 或后台 worker 来构建。

## 产出

`outputs/skill-hybrid-planner.md` 生成一个混合规划器脚手架（HTN 或演化），其中 LLM 的角色被明确限定。

## 练习

1. 给 HTN 规划器增加回溯：当某个算子的后置条件在运行时失败，回滚并尝试下一个方法。
2. 给 ChatHTN 增加 LLM-方法缓存：当 LLM 在状态模式 `P` 下分解任务 `T` 时，存储结果。下次调用时先查方法库。
3. 将演化搜索的评估器替换为真实测试套件。演化一个能通过 20 个测试用例的排序函数；报告收敛所需的代数。
4. 阅读 AlphaEvolve 的评估器设计笔记。为你关心的领域设计一个评估器（SQL 查询优化、测试套件精简、部署 YAML）。
5. 两者结合：用 HTN 将复合任务分解为子任务，然后对每个子任务的原子算子使用演化搜索。哪里表现突出，哪里过度工程了？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| HTN | "层级规划器" | 带算子、前置条件、效果的任务分解 |
| Method | "分解规则" | 将复合任务拆分为子任务的方式 |
| Operator | "原子动作" | 附带前置条件和效果的具象步骤 |
| ChatHTN | "LLM + HTN" | 符号规划器在无方法匹配时询问 LLM |
| AlphaEvolve | "演化代码搜索" | 组合 LLM 变异代码，确定性评估器选优 |
| Fitness 函数 | "评估器" | 对输出进行确定性、可机器校验的打分 |
| 在线方法学习 | "缓存的 LLM 分解" | 存储并泛化 LLM 方案以降低查询成本 |
